# ELKスタック 振動センサー異常検知 まとめ

## 1. 背景・課題

### センサーデータのパターン
- 7日前から振動センサーの値が**徐々に上昇**していく
- 7日後に振動センサーの値が**急激に下落**する

### 問題点
生の振動値を固定閾値で監視するだけでは、**急落部分しか検知できない**。  
上昇トレンド（予兆）の検知が困難。

---

## 2. ELKスタックでの検知手段の検討

### 検討した手段の比較

| 手段 | 費用 | 上昇トレンド検知 | 急落検知 | 実装難易度 |
|------|------|:-:|:-:|:-:|
| Elasticsearch Watcher | 有料（Basic以上） | ○ | ◎ | 中 |
| Kibana Alerting | 無料 | △ | ○ | 低 |
| **ElastAlert2** | **無料** | **◎** | **◎** | **中** |
| Prometheus連携 | 無料 | ◎ | ◎ | 高 |

### 採用方針
- Watcherは追加ライセンスが必要なため除外
- **Kibana 8.17.0のRulesメニュー**から設定する方針に決定

---

## 3. データの見方（検知アプローチ）

生の値だけでなく、データの見方を変えることで上昇トレンドも検知可能。

### 検討した手法

| 手法 | 概要 | おすすめ度 |
|------|------|:-:|
| 移動平均との乖離 | 現在値と移動平均の差分を監視 | ⭐⭐⭐（別途確認済みのため除外） |
| **レンジ（Max-Min）監視** | **一定期間内の最大値と最小値の差を監視** | **⭐⭐⭐** |
| CUSUM（累積和） | 正常値からのズレを累積して監視 | ⭐⭐⭐ |
| 変化率の累積 | 前期間比の変化率を累積して監視 | ⭐⭐ |
| パーセンタイル逸脱 | 過去データの分布から逸脱を監視 | ⭐ |

> 移動平均は別途確認項目として設けているため除外。

---

## 4. レンジ（Max-Min）とは

**一定期間内の「最大値 - 最小値」の幅**のこと。

```
【例：1時間ウィンドウ内の振動値】

上昇中（day1〜day6）：
  day1: 10〜12 → レンジ = 2
  day3: 13〜17 → レンジ = 4
  day6: 25〜32 → レンジ = 7

急落時（day7）：
  振動値 30 → 5 → レンジ = 25 ← 急落でレンジが爆発的に拡大！
```

### なぜ急落を検知できるのか

| 状況 | 最大値 | 最小値 | レンジ |
|------|--------|--------|--------|
| 平常時 | 12 | 10 | 2（小さい） |
| 上昇中 | 30 | 25 | 5（やや大きい） |
| 急落時 | 30 | 5 | **25（異常に大きい）✅** |

急落が起きると**同じ時間窓の中に高い値と低い値が共存する**ため、レンジが急拡大する。  
これを固定閾値（例：レンジ > 15）で検知する。

## 5. Kibana 8.17.0 でのレンジ検知設定

### 制約事項
Kibana 8.17.0の「Elasticsearch query」ルールでは、`bucket_script`による**Max-Minの直接計算がUI上ではサポートされていない**。

### 方法A：Transformを使う方法（2段階アプローチ）

#### 全体フロー
```
vibration-sensor*（生データ）
　　↓
Transform（1時間ごとにMax / Min / Rangeを計算）
　　↓
vibration-sensor-range（レンジ計算済みインデックス）
　　↓
Kibana Rule（vibration_range > 20 で判定）
　　↓
vibration-alerts（アラート記録）
```

#### Step1：Transformの作成
**Stack Management → Transforms → Create Transform**

**Pivot設定（Advanced pivot editor / Edit JSONで入力）：**

```json
{
  "group_by": {
    "timestamp_bucket": {
      "date_histogram": {
        "field": "@timestamp",
        "fixed_interval": "1h"
      }
    }
  },
  "aggregations": {
    "max_vibration": {
      "max": { "field": "vibration" }
    },
    "min_vibration": {
      "min": { "field": "vibration" }
    },
    "vibration_range": {
      "bucket_script": {
        "buckets_path": {
          "max_val": "max_vibration",
          "min_val": "min_vibration"
        },
        "script": "params.max_val - params.min_val"
      }
    }
  }
}
```

**Transformの実行モード：**

| モード | 用途 |
|--------|------|
| Batch | 過去データの一括変換 |
| **Continuous** | リアルタイム検知にはこちらを選択 |

**プレビュー確認イメージ：**

| timestamp_bucket | max_vibration | min_vibration | vibration_range |
|-----------------|--------------|--------------|----------------|
| 2026-03-10 00:00 | 35 | 10 | 25 |
| 2026-03-16 00:00 | 35 | 5 | **30 ← 急落検知** |

#### Step2：Kibana Ruleの作成
**Stack Management → Rules → Create Rule → Elasticsearch query**

**クエリ設定：**
```json
{
  "query": {
    "range": {
      "@timestamp": {
        "gte": "now-1h"
      }
    }
  },
  "aggs": {
    "latest_range": {
      "max": {
        "field": "vibration_range"
      }
    }
  }
}
```

**条件設定：**

| 項目 | 設定値 |
|------|--------|
| Index | `vibration-sensor-range` |
| Time field | `@timestamp` |
| Time window | `1 hour` |
| Threshold | `latest_range` **IS ABOVE** `20` |
| Check every | `1 minute` |

**アクション設定（Elasticsearchへ記録）：**
```json
{
  "rule_name": "{{rule.name}}",
  "detected_at": "{{date}}",
  "vibration_range": "{{context.value}}",
  "message": "振動値のレンジが閾値(20)を超えました"
}
```

### 方法B：Transformを使わない方法（シンプル）

Transformを使わず、**Maxのみで急落を検知**する最もシンプルなアプローチ。

**発想：**
```
通常時のMax：30〜35
急落後のMax： 5

→ Max IS BELOW 10 で検知！
```

**Kibana Rule設定：**

| 項目 | 設定値 |
|------|--------|
| Index | `vibration-sensor*` |
| Rule type | Elasticsearch query |
| Time window | `1 hour` |
| 条件 | `max_vibration` **IS BELOW** `10` |
| Check every | `1 minute` |

**方法の比較：**

| 方法 | Transform不要 | 実装難易度 | 上昇トレンド検知 | 急落検知 |
|----|:-:|:-:|:-:|:-:|
| Transformあり（レンジ） | ❌ | 中 | ○ | ◎ |
| **Transformなし（Maxのみ）** | **✅** | **低** | **△** | **◎** |

## 6. ElastAlert2でのアラート記録

### Elasticsearchへの記録

ElastAlert2はアラート結果を自動的にElasticsearchに記録する。

**自動作成されるインデックス：**

| インデックス名 | 内容 |
|---|---|
| `elastalert_status` | ルールの実行履歴・最終チェック時刻 |
| `elastalert_status_error` | エラーログ |
| `elastalert_alerts` | **発報したアラートの記録** |

**任意インデックスへの書き込み設定：**
```yaml
alert:
  - elastalert_alert

alert_index: my_alerts   # 任意のインデックス名
```

### Kibanaでの可視化

アラートインデックスをData Viewに登録することでダッシュボード表示が可能。

```
Stack Management → Data Views → Create
index pattern: elastalert_alerts*
```

## 7. まとめ・推奨構成

### デモ向け推奨構成（2段階アラート）

```
【予兆アラート（上昇トレンド検知）】
　Transform でレンジを計算
　→ vibration_range が通常の1.5倍を超えたら WARNING

【異常アラート（急落検知）】
　Kibana Rule で Max IS BELOW 固定閾値
　→ max_vibration が 10 を下回ったら CRITICAL
```

### 構成選択ガイド

| 要件 | 推奨手段 |
|------|---------|
| 最もシンプルに急落だけ検知したい | Kibana Rule（Max IS BELOW） |
| 上昇トレンドも含めて検知したい | Transform + Kibana Rule（レンジ監視） |
| より高精度な異常検知が必要 | ElastAlert2（spike type） |
