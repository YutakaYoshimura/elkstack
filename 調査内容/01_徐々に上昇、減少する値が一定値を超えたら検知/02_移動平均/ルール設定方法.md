# ELKスタックにおける移動平均を用いた異常検知

---

## 1. 移動平均とは

「直近N個の値の平均を、時間とともにスライドさせていく」手法。

例）ウィンドウサイズ3の場合：
- 時刻5の移動平均 = （時刻3 + 時刻4 + 時刻5）÷ 3
- 時刻6の移動平均 = （時刻4 + 時刻5 + 時刻6）÷ 3

### なぜ異常検知に使えるのか

移動平均は「その時点での正常な水準」を表す。実測値が移動平均から大きく外れていれば異常のサイン。この乖離を数値化したものが **Z-score**。

```
Z-score = |実測値 − 移動平均| ÷ 標準偏差
```

| Z-score | 確率 | 意味 |
|---|---|---|
| 1以上 | 約32% | 少し外れている |
| 2以上 | 約5% | かなり外れている |
| 3以上 | 約0.3% | ほぼ異常と見なせる |

### ウィンドウサイズの使い分け

- **小さい**（3〜5）：直近の変化に敏感
- **大きい**（10〜30）：長期的なトレンドを反映

## 2. ELKスタックの全体構成

```
Phase 1: 収集・格納
  Logstash（ingest pipeline）→ Elasticsearch（生データ格納）

Phase 2: 異常検知
  Elasticsearch Aggregation（moving_fn / ewma）→ Logstash / script（Z-score判定）→ Elasticsearch（anomalyインデックス）

Phase 3: アラート・可視化
  Alert output（Slack / PagerDuty）／ Kibana（ダッシュボード）
```

---

## 3. Phase 1：Logstash → Elasticsearch（生データ格納）

Logstashはシンプルに生データをそのまま送るだけ。移動平均の計算はES側に任せる。

```ruby
# logstash-ingest.conf
input {
  beats { port => 5044 }
}

filter {
  grok {
    match => { "message" => "%{TIMESTAMP_ISO8601:timestamp} value=%{NUMBER:raw_value}" }
  }
  mutate {
    convert => { "raw_value" => "float" }
  }
  date {
    match => ["timestamp", "ISO8601"]
    target => "@timestamp"
  }
}

output {
  elasticsearch {
    hosts => ["http://localhost:9200"]
    index => "metrics-%{+YYYY.MM.dd}"
  }
}
```

## 4. Phase 2：Elasticsearch Aggregation で移動平均を計算

ElasticsearchのPipeline Aggregationとして `moving_fn`（7.x以降推奨）が組み込まれている。

### 移動平均クエリ（Kibana Dev Toolsで確認）

```json
GET metrics-*/_search
{
  "size": 0,
  "aggs": {
    "per_minute": {
      "date_histogram": {
        "field": "@timestamp",
        "fixed_interval": "1m"
      },
      "aggs": {
        "avg_value": {
          "avg": { "field": "value" }
        },
        "moving_avg": {
          "moving_fn": {
            "buckets_path": "avg_value",
            "window": 10,
            "script": "MovingFunctions.unweightedAvg(values)"
          }
        },
        "moving_std": {
          "moving_fn": {
            "buckets_path": "avg_value",
            "window": 10,
            "script": "MovingFunctions.stdDev(values, MovingFunctions.unweightedAvg(values))"
          }
        },
        "upper_bound": {
          "bucket_script": {
            "buckets_path": {
              "avg": "moving_avg",
              "std": "moving_std"
            },
            "script": "params.avg + params.std * 3"
          }
        },
        "is_anomaly": {
          "bucket_script": {
            "buckets_path": {
              "val": "avg_value",
              "avg": "moving_avg",
              "std": "moving_std"
            },
            "script": "params.std > 0 ? Math.abs(params.val - params.avg) / params.std : 0"
          }
        }
      }
    }
  }
}
```

> **`is_anomaly` はESに組み込まれたキーワードではなく、自分で定義した集計名。**  
> scriptの中身がZ-scoreの計算式そのもの。名前は `z_score` 等に変更しても動作は同じ。

## 5. Phase 3-A：Kibana Rules による異常検知アラート

**Stack Management → Rules and Connectors → Create rule**

### 基本設定

```
Name:        移動平均 異常検知アラート
Check every: 1 minute
Notify:      Only on status change  ← 推奨（異常→正常に戻ったときも通知）
```

### Rule type

```
Elasticsearch query  を選択
```

### クエリ設定

```json
{
  "index": "metrics-*",
  "time_field": "@timestamp",
  "time_window_size": 15,
  "time_window_unit": "m",
  "body": {
    "size": 0,
    "aggs": {
      "per_minute": {
        "date_histogram": {
          "field": "@timestamp",
          "fixed_interval": "1m"
        },
        "aggs": {
          "avg_value": { "avg": { "field": "value" } },
          "moving_avg": {
            "moving_fn": {
              "buckets_path": "avg_value",
              "window": 10,
              "script": "MovingFunctions.unweightedAvg(values)"
            }
          },
          "moving_std": {
            "moving_fn": {
              "buckets_path": "avg_value",
              "window": 10,
              "script": "MovingFunctions.stdDev(values, MovingFunctions.unweightedAvg(values))"
            }
          },
          "is_anomaly": {
            "bucket_script": {
              "buckets_path": {
                "val": "avg_value",
                "avg": "moving_avg",
                "std": "moving_std"
              },
              "script": "params.std > 0 ? Math.abs(params.val - params.avg) / params.std : 0"
            }
          }
        }
      }
    }
  }
}
```

> `time_window_size` を15分以上にしないとウィンドウ10点分のデータが揃わずZ-scoreが計算されないので注意。

### 閾値条件

```
WHEN     aggregation value of  is_anomaly
IS ABOVE  3.0
```

### アクション（通知先）

```
# Slack
Connector: Slack
Message:
  異常検知アラート
  - 時刻: {{date}}
  - Z-score: {{context.value}}

# Email
Connector: Email
To:      ops-team@example.com
Subject: [ALERT] 移動平均 異常検知
Body:    Z-scoreが閾値を超えました。検知時刻: {{date}} / Z-score: {{context.value}}
```

### よくあるつまずきポイント

| 問題 | 原因 | 対処 |
|---|---|---|
| アラートが全く発火しない | Time windowが短すぎる | Time windowを15分以上に広げる |
| 通知が大量に飛ぶ | Notifyが `Every time alert is active` | `Only on status change` に変更 |
| `is_anomaly` が見つからないエラー | aggregation名のタイポ | Dev Toolsでクエリ単体を先に確認 |

## 6. Phase 3-B：Elasticsearch Watcher（より細かい制御が必要な場合）

```json
PUT _watcher/watch/moving_avg_anomaly
{
  "trigger": {
    "schedule": { "interval": "1m" }
  },
  "input": {
    "search": {
      "request": {
        "indices": ["metrics-*"],
        "body": {
          "size": 0,
          "aggs": {
            "per_minute": {
              "date_histogram": {
                "field": "@timestamp",
                "fixed_interval": "1m"
              },
              "aggs": {
                "avg_value": { "avg": { "field": "value" } },
                "moving_avg": {
                  "moving_fn": {
                    "buckets_path": "avg_value",
                    "window": 10,
                    "script": "MovingFunctions.unweightedAvg(values)"
                  }
                },
                "moving_std": {
                  "moving_fn": {
                    "buckets_path": "avg_value",
                    "window": 10,
                    "script": "MovingFunctions.stdDev(values, MovingFunctions.unweightedAvg(values))"
                  }
                },
                "is_anomaly": {
                  "bucket_script": {
                    "buckets_path": {
                      "val": "avg_value",
                      "avg": "moving_avg",
                      "std": "moving_std"
                    },
                    "script": "params.std > 0 ? Math.abs(params.val - params.avg) / params.std : 0"
                  }
                }
              }
            }
          }
        }
      }
    }
  },
  "condition": {
    "script": {
      "source": """
        def buckets = ctx.payload.aggregations.per_minute.buckets;
        if (buckets.size() == 0) return false;
        def last = buckets[buckets.size() - 1];
        def z = last?.is_anomaly?.value ?: 0;
        return z > 3.0;
      """
    }
  },
  "actions": {
    "notify_slack": {
      "throttle_period": "10m",
      "webhook": {
        "method": "POST",
        "url": "https://hooks.slack.com/services/XXX/YYY/ZZZ",
        "body": "{\"text\": \"異常検知: Z-score={{ctx.payload.aggregations.per_minute.buckets[-1].is_anomaly.value}}\"}"
      }
    }
  }
}
```

`throttle_period: 10m`：異常が続いても10分に1回しか通知しないのでアラート過多を防げる。

## 7. SIEM（検出ルール）との連携

SIEMの検出ルールは集計計算ができないため、移動平均による異常検知を直接設定することはできない。

| | SIEM 検出ルール | Rules and Connectors |
|---|---|---|
| KQL / EQL での条件指定 | 可能 | 可能 |
| 移動平均の計算 | **不可** | 可能（Query DSL） |
| Z-scoreによる異常判定 | **不可** | 可能 |
| セキュリティアラートとの統合 | 可能 | 不可 |
| Cases（インシデント管理）との連携 | 可能 | 不可 |

### 回避策：異常データをESに書き戻してSIEMから参照

```
① Rules and Connectors で移動平均の異常検知
        ↓
② 異常イベントを anomalies-* インデックスに書き込む
        ↓
③ SIEM の検出ルールで anomalies-* を監視（KQLで記述可）
```

SIEMルールのKQL例：

```
anomaly: true and z_score > 3.0
```

---

## 8. KQL vs Query DSL

KQLは**検索・絞り込み**専用。移動平均・Z-scoreなどの**集計・計算**はQuery DSLのみ可能。

| | KQL | Query DSL |
|---|---|---|
| できること | フィルタ・絞り込みのみ | フィルタ＋集計＋計算 |
| 書き方 | `value > 10 AND host: "web-01"` | JSONで記述 |
| 移動平均の計算 | 不可 | `moving_fn` で可能 |
| Z-scoreの計算 | 不可 | `bucket_script` で可能 |
