# Podman環境移行・Elasticsearch起動エラー対応まとめ

## 環境
- OS: Windows (PowerShell)
- コンテナ管理: Podman (rootlessモード)
- 構成: ELKスタック (Elasticsearch 8.17.0, Kibana, Logstash, etc.)

---

## 1. エクスポート手順

### イメージのエクスポート
```powershell
$images = podman images --format "{{.Repository}}:{{.Tag}}" | Where-Object { $_ -ne "<none>:<none>" }
podman save $images -o all_images.tar
```

### ボリュームのエクスポート
```powershell
foreach ($vol in (podman volume ls -q)) {
    podman volume export $vol -o "vol_${vol}.tar"
}
```

---

## 2. インポート手順

### 事前準備: Podman VMの起動確認
```powershell
podman machine list
podman machine start  # 停止している場合
```

### イメージのインポート
```powershell
podman load -i all_images.tar
```
> ⚠️ ファイルパスが通らない場合はフルパスで指定する
> 例: `podman load -i C:\Users\ユーザー名\podman_export\all_images.tar`

### ボリュームのインポート
```powershell
foreach ($f in (Get-ChildItem -Path "podman_export\vol_*.tar")) {
    $vol_name = $f.BaseName -replace "^vol_", ""
    podman volume create $vol_name
    podman volume import $vol_name $f.FullName
}
```

> ⚠️ インポート後はボリュームの所有者情報がリセットされる場合がある（後述）

## 3. Elasticsearch起動エラー対応

### エラー内容
```
dependency failed to start: container elasticsearch exited (1)
```
ログの根本原因：
```
java.nio.file.AccessDeniedException: /usr/share/elasticsearch/data/.es_temp_file
write permission is required
```

### 原因
`podman volume export/import` でボリュームを移行した際に、ファイルの所有者情報がリセットされる。
インポート後のボリュームがroot所有になり、Elasticsearchユーザー（UID:1000）が書き込めない状態になる。

### 解決方法

**方法1: コンテナ経由で権限変更（推奨）**
```powershell
podman run --rm `
  --volume elk_es-data:/usr/share/elasticsearch/data `
  --user root `
  docker.elastic.co/elasticsearch/elasticsearch:8.17.0 `
  chown -R 1000:1000 /usr/share/elasticsearch/data
```

**方法2: ボリュームを削除して再作成（データ消去）**
```powershell
podman compose down
podman volume rm elk_es-data
podman volume create elk_es-data
podman compose up -d
```

### 再起動手順
```powershell
podman compose down
podman compose up -d
podman logs elasticsearch
```

## 4. 補足: ボリューム重複について

`podman volume export/import` を複数回実行したり、compose.ymlのボリューム名が変わると重複ボリュームが作成される。

```powershell
# ボリューム一覧確認
podman volume ls
```

| ボリューム名 | 状態 |
|---|---|
| elk_es-data | ✅ 正しい（compose.ymlで使用中） |
| elk_es_data | ❌ 不要（過去に誤って作成） |
| 75d75766e04d... | ❌ 不要（匿名ボリューム） |

不要なボリュームの削除：
```powershell
podman volume rm elk_es_data
podman volume rm 75d75766e04deaffa3243b63e8d137b76560a4eae372275cb043cfb058affbce
```

## 5. 注意事項

| 項目 | 内容 |
|---|---|
| Podman VM起動 | PC再起動後は毎回 `podman machine start` が必要 |
| ボリューム権限 | インポート後は必ずコンテナ経由でchownを実行する |
| ボリューム名 | compose.ymlのボリューム名とインポート時の名前を一致させる |
| PowerShell構文 | Bashのforループは使えないため `foreach` に書き換える |
