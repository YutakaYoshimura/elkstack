## Day 1〜2: データの流れを「設定ファイル」から読む

### 今日の目的

「データはどこから来て、どこへ行くのか」を、**設定ファイルを読んで** 理解する。
Kibana の画面を見る前に、仕組みをファイルベースで把握することが大切。

---

> ### コラム: ELKスタックとはどんな仕組みか
>
> ELK スタックは 3 つのツールの頭文字をとった名前。
>
> ```
> センサーデータ（CSV）
>      ↓
> Logstash（L）: データを読み込んで、加工して、送る「パイプライン」ツール
>      ↓
> Elasticsearch（E）: データを格納・検索するデータベース
>      ↓
> Kibana（K）: データを可視化するダッシュボードツール
> ```
>
> 今回の構成では、これに加えて:
> - **ElastAlert2**: Elasticsearch のデータを定期的に監視してアラートを発報するツール
> - **Python ML コンテナ**: より高度な機械学習の実行環境
>
> 全体像を頭に入れてから作業すると、何をしているか迷いにくくなる。

---

### やること

#### 1. Logstash のパイプライン設定ファイルを読む

ファイルを開く: `logstash/pipeline/sensor-data.conf`

このファイルは Logstash の「パイプライン定義ファイル」。
**CSV ファイルのデータを Elasticsearch に取り込むための手順書** であり、
「何を読んで、どう加工して、どこへ出力するか」が書いてある。

> **補足: 取り込み先の Elasticsearch とは何か？**
>
> Elasticsearch は「データベース」の一種だが、MySQL や PostgreSQL のような RDB（リレーショナルデータベース）とは設計思想が異なる。
> RDB との対比で理解するとわかりやすい:
>
> | RDB の概念 | Elasticsearch の概念 | 今回の例 |
> |---|---|---|
> | データベース | クラスタ | `predictive-maintenance` |
> | テーブル | **インデックス** | `sensor-data-2026.01` |
> | 行（レコード） | ドキュメント | センサーデータ1件 |
> | 列（カラム） | フィールド | `vibration`, `temperature` 等 |
> | スキーマ定義（CREATE TABLE） | **マッピング** | フィールドの型情報 |
>
> RDB では事前に `CREATE TABLE` でテーブル定義が必要だが、
> Elasticsearch では **最初のデータを送り込んだ時点でインデックスとマッピングが自動生成** される（「動的マッピング」と呼ぶ）。
> output セクションで指定しているのは「どのインデックス名に送るか」だけで、テーブル定義に相当する作業は不要。

---

> ### コラム: Logstash のパイプラインの 3 セクション
>
> Logstash の設定ファイルは必ず以下の 3 セクションで構成される。
>
> ```
> input {
>   # どこからデータを読み込むか
>   # 例: CSV ファイル、Kafka、MQTT など
> }
>
> filter {
>   # データをどう加工するか
>   # 例: 列名を変換する、型を変換する、フィールドを追加する
> }
>
> output {
>   # データをどこへ送るか
>   # 例: Elasticsearch、別のファイル、標準出力
> }
> ```
>
> この 3 段階の流れを「パイプライン」と呼ぶ。
> 水道管のようにデータが流れていくイメージ。

---

---

> ### コラム: `sincedb_path` とは何か（実例で理解する）
>
> Logstash はファイルを読み込むとき、「どこまで読んだか」を sincedb ファイルに記録する。
> これにより、再起動しても続きから読み込める。
>
> **sincedb が有効な場合（本番環境の典型例）**:
> ```
> 1日目の朝: sensor_data.csv に 1,000行ある
>   → Logstash が 1,000行読み込む
>   → sincedb に「1,000行目まで読んだ」と記録
>
> 1日目の夜: 新しいデータが追記され 1,500行になる
>   → sincedb を見て「1,000行目まで読み済み」と判断
>   → 1,001〜1,500行目の 500行だけ読み込む（重複なし）
> ```
>
> **sincedb が無効な場合（今回の設定: `sincedb_path => "/dev/null"`）**:
> ```
> Logstash 起動: 38,880行全部読む
> Logstash 再起動: また38,880行全部読む ← 毎回最初から
> ```
>
> 今回は「完成済みの CSV を一括投入する検証環境」なので、毎回先頭から読んでくれた方が都合がよい。
> 本番でリアルタイムにデータが追記されるケースでは sincedb を有効にして重複を防ぐのが一般的。

> ### コラム: Elasticsearch の REST API とは
>
> Elasticsearch はデータの操作をすべて **HTTP リクエスト**（REST API）で行う。
>
> API エンドポイントの構造:
> ```
> http://localhost:9200 / インデックス名 / アクション ? パラメータ
>
> 例: http://localhost:9200/sensor-data-*/_search?size=1&pretty
>
> localhost:9200      → Elasticsearch のアドレス（ローカル環境）
> sensor-data-*       → 対象のインデックス（* はワイルドカード）
> _search             → 検索アクション
> size=1              → 最大 1 件だけ返す
> pretty              → JSON を見やすく整形して返す
> ```
>
> API の結果を見ると、Elasticsearch がどんな形でデータを保存しているかがわかる。

---

API の結果 JSON を見て確認する:

```
Q: `_source` の中に何のフィールドがあるか？列挙してみる。
Q: `vibration`、`temperature`、`current` の値の「型」は何か？
   （数値として扱えているか、文字列になっていないか）
Q: `@timestamp` フィールドはどんな形式で保存されているか？
Q: `device_id` フィールドの値は何か？
```

---

> ### コラム: Elasticsearch のデータ型と mapping
>
> Elasticsearch はデータを **インデックス** に格納し、各フィールドに **型** がある。
>
> よく使う型:
> | 型 | 用途 | 例 |
> |---|---|---|
> | `float` / `double` | センサーの数値 | vibration: 3.21 |
> | `keyword` | 完全一致検索用の文字列 | device_id.keyword: "pump-01" |
> | `text` | 全文検索用の文字列 | （今回はあまり使わない） |
> | `date` | 日時 | @timestamp: "2026-01-01T00:00:00Z" |
>
> **重要**: `mutate { convert }` で型変換をしないと、数値が文字列として保存される。
> 文字列のままでは「振動が 3.21 以上のデータを絞り込む」ような集計ができない。
> これが Logstash の filter セクションで型変換している理由。

---

> ### コラム: `text` 型と `keyword` 型の違い（重要）
>
> 文字列フィールド（`device_id` 等）には、Elasticsearch が自動的に `text` と `keyword` の両方を作る。
>
> | 型 | 保存方法 | 検索例 | 用途 |
> |---|---|---|---|
> | `text` | 文字列を分解して保存（"ポンプA-01" → "ポンプ", "A", "01"） | 部分一致検索 | ブログ記事やログの全文検索 |
> | `keyword` | 文字列をそのまま保存（"ポンプA-01" → "ポンプA-01"） | 完全一致検索 | ID・カテゴリ名での絞り込み |
>
> **keyword 型でしかできないこと**:
> - 完全一致検索（`device_id.keyword : "pump-01"`）
> - 正しいソート（五十音順等）
> - 正しい集計（設備ごとの件数・平均等）
>
> text 型で集計すると「ポンプ」「A」「01」がバラバラに集計され壊れた結果になる。
> KQL で `.keyword` を付けるのは「keyword 型を使って完全一致検索する」ため。

---

> ### コラム: 動的マッピングと既存データの注意点
>
> Elasticsearch は最初のデータ投入時にマッピング（型定義）を自動生成する（「動的マッピング」）。
> RDB のように事前の `CREATE TABLE` は不要だが、**一度作られたフィールドの型は変更できない**。
>
> | 操作 | できる？ | 方法 |
> |---|---|---|
> | フィールド追加 | 簡単 | 新フィールド付きデータを送るだけ（既存データには追加されない） |
> | フィールドの型変更 | **できない** | インデックスの再作成（reindex）が必要 |
>
> フィールドが存在しないドキュメントを検索・集計しても **エラーにはならず、単にスキップされる**。
> これは Elasticsearch の設計思想: **「あるものだけ使う、ないものは無視する」** が基本。
> RDB の「全行が同じカラム構成」という前提とは異なるので注意。

## Day 3〜4: 正常期間のデータを「統計的に」調べ、仮の閾値を考える

### 今日の目的

Week 3 の「閾値をどう決めるか」の核心演習に備えて、**正常時の振動値の特性を数字で把握する**。

「直感や目視で閾値を決める」のと「データから計算して決める」のは全く違う。
今日は後者の基礎力を身につける。

---

> ### コラム: なぜ「正常データを分析してから」閾値を決めるのか
>
> 閾値を設定するとき、よくある（良くない）やり方:
>
> ❌ 「振動値は大体 3〜4 くらいだから、6.0 を超えたら異常にしよう」
>
> この決め方の問題点:
> - 「大体 3〜4 くらい」が正確かどうかわからない
> - 「なぜ 6.0 なのか」をデータで説明できない
> - 設備が変わったときに使い回せない
>
> ✅ 正しい決め方: 正常期間のデータから統計を計算し、根拠をもって閾値を設定する
>
> 「正常期間の avg（3.2）に std（0.4）の3倍を足した 4.4 を閾値にした。
>  これは正常データの 99.7% がこの範囲に収まる統計的な上限値である」
>
> この説明は誰でも再現でき、後から検証できる。

---

### やること

#### 1. Kibana で正常期間の振動値を「目で見る」

1. Kibana を開く: http://localhost:5601（elastic / changeme でログイン）
2. 左メニュー → Discover
3. データビュー（インデックスパターン）: `sensor-data` を選択
4. 右上の時間範囲を「2026-01-01」〜「2026-01-12」に設定
5. KQL フィルターを入力:

```

```

---

> ### コラム: KQL（Kibana Query Language）の基本
>
> KQL は Kibana の検索バーで使えるクエリ言語。
>
> よく使う構文:
>
> | 用途 | 構文 | 例 |
> |---|---|---|
> | 完全一致 | `フィールド名 : "値"` | `device_id.keyword : "pump-01"` |
> | 数値の範囲 | `フィールド名 >= 値` | `vibration >= 4.0` |
> | AND 条件 | `条件1 and 条件2` | `device_id.keyword : "pump-01" and vibration >= 4.0` |
> | OR 条件 | `条件1 or 条件2` | `device_id.keyword : "pump-01" or device_id.keyword : "motor-01"` |
> | NOT 条件 | `not 条件` | `not device_id.keyword : "compressor-01"` |
>
> `.keyword` を付けるタイミング:
> - `device_id.keyword` のように **完全一致で文字列検索するとき** は `.keyword` が必要
> - `vibration` のように **数値フィールド** には不要

#### 2. Elasticsearch API で正常期間の統計を計算する

Discover の目視だけでは「平均がいくらか」「ばらつきはどのくらいか」がわからない。
API を使って数値として確認する。

```powershell
# 正常期間（1/1〜1/12）の pump-01 振動の統計
curl.exe -u elastic:changeme -X POST `
  "http://localhost:9200/sensor-data-2026.01/_search?pretty" `
  -H "Content-Type: application/json" `
  -d '{
    "size": 0,
    "query": {
      "bool": {
        "filter": [
          {"term": {"device_id.keyword": "pump-01"}},
          {"range": {"@timestamp": {"gte": "2026-01-01", "lt": "2026-01-13"}}}
        ]
      }
    },
    "aggs": {
      "vibration_stats": {
        "extended_stats": {"field": "vibration"}
      }
    }
  }'
```

---

> ### コラム: Elasticsearch の集計（Aggregation）とは
>
> `_search` API では検索だけでなく、データの集計（集約）もできる。
> `aggs`（aggregations の略）セクションに集計の定義を書く。
>
> `extended_stats` は複数の統計量をまとめて計算してくれる集計。
>
> ```json
> {
>   "aggregations": {
>     "vibration_stats": {
>       "count":  1728,         ← データ件数
>       "min":    2.45,         ← 最小値
>       "max":    4.83,         ← 最大値
>       "avg":    3.21,         ← 平均値
>       "sum":    5546.88,      ← 合計値
>       "variance": 0.1444,     ← 分散
>       "std_deviation": 0.38,  ← 標準偏差（σ）
>       "std_deviation_bounds": {
>         "upper": 3.97,        ← avg + 2σ
>         "lower": 2.45         ← avg - 2σ
>       }
>     }
>   }
> }
> ```
>
> `size: 0` を指定すると、検索結果のドキュメント（実データ）は返さず、
> 集計結果だけを返す。大量データの統計を効率よく取得できる。
>
> **`avg + 3σ` の計算（API では `upper` は avg+2σ まで）**:
> `avg + 3σ` は手計算で求める:
> `avg + 3σ = avg + 3 × std_deviation`

## Day 5〜7: 単一センサー（振動のみ）のダッシュボードを作る

---

> ### コラム: Kibana Lens とは
>
> Kibana には複数の可視化ツールがあるが、現在は **Lens** が推奨の標準ツール。
>
> Lens の特徴:
> - ドラッグ&ドロップでグラフを作れる
> - グラフの種類を後から変更しやすい
> - X軸・Y軸・集計方法を直感的に設定できる
>
> 基本操作の流れ:
> 1. Kibana → Visualize Library → Create visualization → Lens を選択
> 2. 左パネル: データビューを選択（sensor-data）
> 3. 左パネルからフィールドをドラッグして軸にドロップ
> 4. 右上の集計方法（平均/最大/最小）を変更
> 5. 右上の Save → ダッシュボードに追加

> ### コラム: ダッシュボードの「フィルター」と「時間範囲」の関係
>
> Kibana ダッシュボードには 2 種類のフィルター制御がある。
>
> 1. **ダッシュボード全体のフィルター**: 右上の時間範囲、または KQL 検索バー
>    - ダッシュボード上の全パネルに影響する
>
> 2. **パネル個別のフィルター**: Lens 編集画面の Filter オプション
>    - そのパネルだけに影響する
>    - `device_id.keyword : "pump-01"` のように特定設備に絞るときに使う
>
> 使い分けの例:
> - 「全設備の比較表」にはダッシュボード全体フィルターを使わない
> - 「pump-01 だけの詳細グラフ」はパネル個別フィルターを使う
