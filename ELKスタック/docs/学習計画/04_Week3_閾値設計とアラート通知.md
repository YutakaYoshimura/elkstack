## Day 15〜16: ElastAlert2 の設定ファイルを「読み解く」

### 今日の目的

閾値アラートを実装する前に、ElastAlert2 がどんなツールで、何を設定しているかを理解する。

---

> ### コラム: ElastAlert2 とは何か
>
> ElastAlert2 は Elasticsearch を監視して、条件に合うデータが見つかったら
> メール・Slack などに通知するツール。
>
> 動作の仕組み:
> ```
> ElastAlert2 の動作（ループ処理）:
>
> 1. 設定ファイル（config.yaml）を読む
>    → どの Elasticsearch に接続するか
>    → どのルールファイルを使うか
>
> 2. ルールファイル（rules/*.yaml）を読む
>    → どのインデックスを監視するか
>    → どんな条件でアラートするか
>    → アラートは何分間隔で確認するか
>
> 3. 定期的に Elasticsearch を検索する（デフォルト 1 分ごと）
>    → 条件に合うデータがあれば通知を送る
>    → 条件に合わなければ次の確認まで待つ
> ```
>
> **ElastAlert2 の役割は「Elasticsearch のウォッチドッグ（番犬）」**:
> データを自分では加工しない。ただ「設定した条件に合うデータが来たら吠える（通知する）」だけ。

---

> ### コラム: ElastAlert2 の主要なルールパラメータ
>
> ```yaml
> name: "ルール名"
> # このルールの識別名。ログとアラートメールの件名に使われる。
>
> type: any
> # ルールの種類。
> # any = 条件に合うデータが 1 件でもあればアラート
> # frequency = 指定時間内に指定件数以上ヒットしたらアラート（詳細は問題1模範解答参照）
>
> index: sensor-data-*
> # 監視対象のインデックスパターン。
> # * はワイルドカードで、sensor-data-2026.01, sensor-data-2026.02 など全月を対象にする。
>
> filter:
>   - term:
>       device_id.keyword: "pump-01"
>   # 完全一致フィルター。device_id が pump-01 のデータのみを対象にする。
>
>   - range:
>       anomaly_score:
>         gte: 0.7
>   # 数値範囲フィルター。anomaly_score が 0.7 以上のデータのみを対象にする。
>   # gte = Greater Than or Equal（以上）
>
> realert:
>   minutes: 30
> # 同じルールのアラートを 30 分間は再送しない（通知の氾濫を防ぐ「チャタリング防止」）
>
> alert:
>   - debug
> # アラートの通知方法。
> # debug = コンソール出力のみ（メールは送らない）。テスト時に使う。
> # email = メール送信。本番時に使う。
> ```
>
> **`realert` の重要性**:
> `realert` を設定しないと、条件を満たすデータが 1 件あるたびに通知が送られる。
> センサーデータが 10 分間隔なら、1 時間で 6 通のメールが届くことになる。
> これが「アラート疲れ」を引き起こす。
>
> **`buffer_time` とは**:
> ElastAlert2 が「今から何分前までのデータを検索するか」の設定。
> リアルタイム監視では最新データの取り込みに若干の遅延があるため、
> 少し過去（デフォルト 15 分）まで遡って確認する。

#### 3. `elastalert-test-rule` を実行してログを確認する

ElastAlert2 にはルールをテスト実行するコマンドが用意されている。
実際にルールがどのように動作するかを、過去のデータで確認できる。

**テスト実行前の確認（重要）**:

`anomaly-alert.yaml` は `prediction-results` インデックス（Python ML の推論結果）を監視している。
このインデックスは `batch_inference.py` を実行しないと存在しない。
テスト前に以下で確認する:

```powershell
# prediction-results インデックスが存在するか確認
curl.exe -u elastic:changeme "http://localhost:9200/prediction-results/_count?pretty"
```

**結果の解釈**:
- `"count": 数値` が返れば、インデックスが存在する → テスト実行へ進む
- `index_not_found_exception` が返った場合 → 以下を実行してデータを生成する:

```powershell
# Python ML の推論を実行して prediction-results インデックスを生成する
docker exec python-ml python batch_inference.py

# 完了後、再度件数を確認
curl.exe -u elastic:changeme "http://localhost:9200/prediction-results/_count?pretty"
```

データが存在することを確認してからテスト実行へ進む。

```powershell
# 既存のルール（anomaly-alert.yaml）をテスト実行する
# 故障1の前後（1/14〜1/21）を対象に
docker exec elastalert elastalert-test-rule `
  --start "2026-01-14T00:00:00" `
  --end "2026-01-21T00:00:00" `
  /opt/elastalert/rules/anomaly-alert.yaml
```

---

> ### コラム: `docker exec` の意味
>
> ElastAlert2 は Docker コンテナの中で動いている。
> そのため ElastAlert2 のコマンドを実行するには「コンテナの中に入って実行する」必要がある。
>
> `docker exec [コンテナ名] [コマンド]` の形で使う。
>
> ```
> docker exec elastalert elastalert-test-rule ...
>            ↑            ↑
>        コンテナ名    コンテナ内のコマンド
> ```
>
> **ファイルパスについて**:
> - ローカルのファイルパス: `elastalert/rules/anomaly-alert.yaml`
> - コンテナ内のパス: `/opt/elastalert/rules/anomaly-alert.yaml`
>
> `docker-compose.yml` でローカルの `elastalert/rules/` フォルダが
> コンテナ内の `/opt/elastalert/rules/` に「マウント（共有）」されているため、
> コマンドではコンテナ内のパスを使う。

> ### コラム: ElastAlert2 の YAML ファイル作成で注意すること
>
> 問題1の Step 3 で YAML ファイルを作成するときの注意点をまとめる。
>
> **YAML の書き方ルール**:
>
> 1. **インデントはスペース 2 つ**（タブ文字は絶対に使わない）
>    ```yaml
>    filter:
>      - term:         ← スペース 2 つ
>          device_id.keyword: "pump-01"   ← スペース 4 つ
>    ```
>
> 2. **コロン（:）の後にスペースが必要**
>    - 正しい: `name: "テスト"`
>    - 間違い: `name:"テスト"`
>
> 3. **文字列はダブルクォートで囲む**（数値は不要）
>    - 文字列: `device_id.keyword: "pump-01"`
>    - 数値: `gte: 4.4`
>
> **YAML エディタでの確認方法**:
> VS Code を使っている場合、`YAML` 拡張機能をインストールすると
> インデントエラーをリアルタイムで検出してくれる。
>
> **よくあるエラーと対処法**:
>
> | エラーメッセージ | 原因 | 対処 |
> |---|---|---|
> | `mapping values are not allowed here` | コロンの後にスペースがない | `: ` にする |
> | `found a tab` | タブ文字が混入 | スペースに置換 |
> | `could not find expected ':' ` | キーの書き方が間違い | YAML 構文を確認 |
> | `ConnectionError` | Elasticsearch が起動していない | `docker compose ps` で確認 |
> | `No hits` (全期間 0 件) | 閾値が高すぎる or device_id が間違い | 値を下げるか、スペルミスを確認 |

## Day 20〜21: 実際にメールを届ける

### 今日の目的

「理論で閾値を設計した」から「実際に通知を受け取る」まで、一連の流れを完成させる。
「自分が設計した閾値でメールが届く」体験が、学習の大きな達成感になる。

---

> ### コラム: Gmail のアプリパスワードとは
>
> Google アカウントの 2 段階認証が有効な場合、通常のパスワードでは
> アプリ（ElastAlert2 など）からメールを送ることができない。
> 「アプリパスワード」は特定のアプリ専用に生成する 16 桁のパスワード。
>
> **取得手順**:
> 1. Google アカウント（accounts.google.com）にログイン
> 2. セキュリティ → 2 段階認証プロセス（有効化されていることを確認）
> 3. 2 段階認証プロセスのページ → 下部の「アプリパスワード」を選択
> 4. アプリを選択: 「その他（任意の名前）」→「ElastAlert2」と入力
> 5. 生成された 16 桁のパスワードを記録する
>
> **注意**: アプリパスワードは他の人に見せないこと。
> また YAML ファイルにそのまま書き込むのではなく、専用の認証ファイルを使う（手順は後述）。

### やること

#### 1. Gmail のアプリパスワードを取得する

上記コラムの手順で取得する。
2 段階認証が有効でない場合は先に有効化する必要がある。

#### 2. SMTP 認証ファイルを作成する

ElastAlert2 は SMTP の認証情報を別ファイル（`smtp_auth.yaml`）に分離して管理する。
パスワードをルールファイルに直接書かないための仕組み。

ファイルのパス: `elastalert/smtp_auth.yaml`

```yaml
user: "自分のGmailアドレス@gmail.com"
password: "xxxxxxxxxxxxxxxx"    # アプリパスワード（スペースなしで 16 桁）
```

> **セキュリティ注意**: このファイルはパスワードを含むため、Git に commit しないこと。
> `.gitignore` に `elastalert/smtp_auth.yaml` が追加されているか確認すること。

#### 3. Docker コンテナから smtp_auth.yaml を読めるように確認する

`docker-compose.yml` で `elastalert/` ディレクトリが
コンテナの `/opt/elastalert/` にマウントされていることを確認する。

```powershell
# docker-compose.yml でマウント設定を確認
# elastalert サービスの volumes セクションを見る（PowerShell）
Select-String -Path docker-compose.yml -Pattern "elastalert:" -Context 0,5
```

マウントされていれば、コンテナ内のパスは `/opt/elastalert/smtp_auth.yaml` になる。

#### 4. ルールファイルにメール設定を追加する

問題1で作成した `elastalert/rules/vibration-threshold-my.yaml` を編集して、
メール通知の設定を追加する。

```yaml
name: "pump-01 振動閾値アラート（自分の設計）"
type: frequency
index: sensor-data-*
num_events: 3
timeframe:
  minutes: 15

filter:
  - term:
      device_id.keyword: "pump-01"
  - range:
      vibration:
        gte: 4.4    # ← 自分が決めた値（または模範解答の avg+3σ）

realert:
  minutes: 60

# ↓ ここからメール設定を追加する
alert:
  - email
email:
  - "自分のメールアドレス@gmail.com"
smtp_host: "smtp.gmail.com"
smtp_port: 587
smtp_auth_file: /opt/elastalert/smtp_auth.yaml

alert_subject: "【振動異常アラート】{0} で振動上昇を検知"
alert_subject_args:
  - device_id

alert_text: |
  【振動異常アラート】
  設備ID:   {0}
  振動値:   {1}
  検知時刻: {2}

  ─────────────────────────────
  閾値: {3}（統計的閾値: 正常期間 avg+3σ）
  ─────────────────────────────

  ↓ Kibana でデータを確認する
  http://localhost:5601

alert_text_args:
  - device_id
  - vibration
  - "@timestamp"
  - "4.4"
```

---

> ### コラム: アラートメールの本文設計の考え方
>
> 現場担当者がメールを受け取ったとき、「何をすべきか」をすぐに判断できる内容にする。
>
> 含めるべき情報:
> - **何が起きているか**: 設備名・センサー名・閾値超過の事実
> - **どのくらいの値か**: 実際の計測値（閾値と比較できるようにする）
> - **いつ起きたか**: 検知日時
> - **どこで確認できるか**: Kibana の URL
> - **なぜアラートが出たか**: 閾値の根拠（設定担当者への信頼性のため）
>
> 含めない方が良い情報:
> - 技術的すぎる詳細（現場担当者には不要）
> - 長すぎる本文（スマホで確認することを想定する）
>
> **「アラートを受け取った担当者は何をするか？」を想像して設計する**
