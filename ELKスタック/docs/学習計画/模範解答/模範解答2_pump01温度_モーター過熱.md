**実装上の課題**:

事前に計算して Elasticsearch に保存する必要がある。
この発想は「発展課題」として取り組む価値がある。

## 問題2の核心: なぜ「同じアプローチ」が通用しにくいのか

温度センサーには**振動センサーと異なる特性**がある。

**振動センサーの特性**（問題1）:
- 正常時は設備の稼働状態に関係なく、ほぼ一定の水準で安定している
- 「正常値の絶対量」が比較的安定 → avg+3σ による絶対値閾値が有効

**温度センサーの特性**（問題2）:
- 温度は「設備にかかる負荷（電流）」と連動して変化する
- 昼間（高負荷）は温度が高く、夜間（低負荷）は低い
- 「正常値の絶対量」が時間帯・負荷によって変動する

### 視覚的なイメージで理解する

```
振動センサーの正常値（時間軸）:
3.1, 3.2, 3.0, 3.1, 3.2, 3.3, 3.1, 3.2, 3.0, 3.2 ...
→ ほぼ水平な帯状に分布する
→ avg+3σ の「水平な境界線」が有効


温度センサーの正常値（時間軸）:
44, 45, 47, 50, 53, 54, 52, 50, 47, 45, 43, 42 ...
   低負荷  →    高負荷ピーク    →    低負荷
→ 時間帯によって上下する「波形」状に分布する
→ avg+3σ の「水平な境界線」では、高負荷時の正常値も超えてしまう
```

### なぜ誤検知が起きるか

**根本原因: 夜間の低温データが avg を引き下げる**

```
正常期間（2026-01-22〜2026-02-08）の温度データの内訳（イメージ）:

  時間帯          温度の水準
  夜間（0〜8時）  約 42〜46 ℃（低負荷・冷却中）
  昼間（8〜18時） 約 50〜54 ℃（高負荷・稼働中）
  夜間（18〜24時）約 45〜48 ℃（徐々に冷却）
```

昼夜両方を含む全期間で avg を計算すると:
- 夜間の低温データが avg を引き下げる（約 47.5 になる）
- avg+3σ（≈ 55.9）は「夜間基準の上限値」になってしまう

```
avg+3σ（約 55.9）を使った場合:
  正常な高負荷運転中（昼間）→ temperature が 54〜56 付近に達することがある
  → 55.9 を超えることがある
  → 正常なのにアラートが出る（誤検知）
```

「なぜ誤検知が起きるか」を一言で:

> **avg+3σ が「全時間帯の平均基準」で計算されているため、
> 昼間の正常な高温が閾値を超えてしまう**

問題1（振動センサー）では昼夜差がほとんどなかったため、この問題が顕在化しなかった。
温度センサーのように「時間帯・負荷によって正常値の水準が変わるセンサー」では、
単純な avg+3σ では不十分になるケースがある。

しかし閾値を高くして誤検知を避けようとすると:

```
閾値を 60 に上げた場合:
  過熱による故障の予兆として温度が 57〜59 まで上昇
  → 60 に達しないためアラートが出ない（検知漏れ）
```

これが温度センサーにおける「閾値設計のジレンマ」。

## センサーの特性を調べてから閾値を設計する: Step 0 の重要性

閾値設計の前に必ず行うべきこと:

**「このセンサーの値は何によって変動するか」を確認する**

温度センサーであれば:
- 電流（負荷）と相関があるか → ある（今回のケース）
- 季節・外気温と相関があるか → 工場内設備なら影響は小さいことが多い
- 時間帯（稼働スケジュール）と相関があるか → あることが多い

確認方法:
1. Kibana の「Discover」で temperature と current を同時に可視化する
2. 電流が上がったとき温度も上がるかを目視で確認する
3. 正常期間の scatter plot（散布図）で相関を確認する

この確認ステップを省くと、閾値設計後に「なぜ誤検知が出るのか」がわからなくなる。

## 3つのアプローチと評価

### アプローチ A: avg+3σ をそのまま適用（問題1と同じ手法）

**閾値**: `temperature > 55.9`

| 指標 | 結果 |
|---|---|
| 最初の検知 | 故障の数日前 |
| 誤検知（正常期間、高負荷時） | 発生する可能性あり |

**評価**: 一定の効果はあるが、高負荷時の誤検知リスクがある。
「なぜ誤検知が出るか」を説明できる → これが問題2の学びポイント。

---

### アプローチ B: 電流と温度の比率を監視する

**考え方**: 「電流が高い → 温度が高い」は正常。「電流が低いのに温度が高い」が異常。

```
異常の指標: temperature / current の値が正常範囲を超えたとき
正常時の比率: 約 ○○（各自の API 実行結果で確認）
閾値: 比率 > ○○
```

**評価**: 原理的に最も正確。負荷変動の影響を受けない。
ただし ElastAlert2 単体では「フィールド間の計算」ができないため、
Logstash で `temp_current_ratio` フィールドを事前計算する必要がある。

**なぜこれが「正確」か**:

```
正常な高負荷運転:
  current = 12A, temperature = 54°C
  比率: 54 / 12 ≈ 4.5

正常な低負荷運転:
  current = 6A, temperature = 27°C
  比率: 27 / 6 = 4.5

→ 負荷が変わっても比率はほぼ一定 → これが「正常の基準」

モーター過熱の予兆:
  current = 8A, temperature = 52°C
  比率: 52 / 8 = 6.5 ← 正常比率（4.5）を大きく超えている

→ 電流は普通なのに温度が異常に高い → アラート！
```

### アプローチ C: avg+3σ + frequency + 高めのマージン（実用的な妥協案）

```yaml
type: frequency
num_events: 5          # 問題1より条件を厳しくする（5件/15分）
timeframe:
  minutes: 15
filter:
  - term:
      device_id.keyword: "pump-01"
  - range:
      temperature:
        gte: 57.0      # avg+3σ より少し高め（誤検知を抑制）
realert:
  minutes: 120         # 温度は変化が遅いため長めに
```

| 指標 | 結果 |
|---|---|
| 最初の検知 | 故障の 3〜4 日前 |
| 誤検知（正常期間） | ほぼ 0 件 |

**評価**: 完璧ではないが、ElastAlert2 の範囲内で実用的に動く設計。
「精度よりも実装の現実性を優先した」という判断を、根拠とともに説明できることが重要。

**なぜ `num_events: 5` に増やすか**:

温度は振動と比べて「変化が緩やか」な特性がある。
- 振動: 異常が始まると数分で閾値を超えることがある
- 温度: 徐々に上昇するため、閾値を超えてから長時間継続する

つまり「温度が閾値を超え続ける」という状況は「本物の異常」である可能性が高い。
条件を `num_events: 5` と厳しくしても、本物の異常は十分な件数が積み上がる。
一方、ノイズや一時的な高負荷による高温は短時間で収まるため、5件に届かない。

**なぜ `realert: minutes: 120` にするか**:

`realert` はアラートが出た後、次のアラートが出るまでの最低間隔。
温度の異常は「急激な変化」よりも「徐々に悪化」するパターンが多い。
1時間ごとにアラートが出ても担当者の対応が追いつかないため、2時間間隔に設定する。

## 模範解答の最終設計（実用的な選択）

```yaml
name: "pump-01 温度異常（高めの閾値 + frequency）"
type: frequency
index: sensor-data-*
num_events: 5
timeframe:
  minutes: 15
filter:
  - term:
      device_id.keyword: "pump-01"
  - range:
      temperature:
        gte: 57.0    # avg+3σ（55.9）より少し高め。根拠: 高負荷時の誤検知を抑制するため
realert:
  minutes: 120
alert:
  - debug
```

**この設計の根拠**:
- 57.0 = avg+3σ（55.9）に 1.1 のマージンを加えた値
- マージンの根拠: 正常期間の高負荷時の最大値（約 54.0）から 3.0 の余裕を持たせた
- frequency（5件/15分）: 温度は変化が緩やかなため、条件を厳しくしても早期検知に影響が少ない
- realert（120分）: 温度異常は長時間継続するため、アラート頻度を抑制

## 問題2から得るべき重要な気づき

### 1. 「センサーの特性」を調べてから閾値を決める

閾値を決める前に「このセンサーの正常値は何によって変動するか」を
必ず確認すること（Step 0 の観察）。

振動は安定している → 絶対値閾値が有効
温度は負荷依存 → 絶対値閾値だけでは不十分 → 追加の工夫が必要

**「先にデータを観察する」という習慣**:
いきなり閾値設計に入るのではなく、まず「このデータはどんな形をしているか」を
グラフや統計値で確認するクセをつけること。
これは予知保全に限らず、データ分析全般で重要な習慣。

### 2. 完璧な手法は「実装コスト」との兼ね合いで決まる

理想的には「電流と温度の比率を監視する」が最も正確。
しかし ElastAlert2 + Logstash の改修が必要 → 実装コストが高い。

今使えるツールの範囲内で「最善」の設計をすることも、実務スキルの一つ。
「完璧な設計」を追い求めて実装が止まるよりも、
「実用的な設計」を実装して運用しながら改善する方が現場では価値がある。

### 3. 設計の根拠をドキュメントに残す

なぜ 57.0 にしたのか、なぜ 55.9 でなくマージンを加えたのか。
これを記録しておかないと、半年後に「なぜこの閾値か」を説明できなくなる。

**残すべき内容の例**:

```
設備: pump-01
センサー: 温度（temperature）
正常期間: 2026-01-22〜2026-02-08
算出方法: 正常期間の avg（47.5）+ 3σ（2.8 × 3 = 8.4）= 55.9
特性上の課題: 電流（負荷）と連動して温度が変動する。高負荷時に 54 度付近まで上昇することがある。
そのため avg+3σ（55.9）に 1.1 のマージンを加えて 57.0 を閾値とした。
frequency: 5件/15分（高負荷時の一時的な高温を誤検知しないようにするため）
設定日: 2026-XX-XX  設定者: ○○
今後の改善案: Logstash で temp/current 比率フィールドを計算し、比率ベースの監視に移行する
```

## 誤検知が起きるとどうなるか: 温度センサー特有のシナリオ

温度の誤検知が具体的にどんな問題を引き起こすかを想像してみる。

**シナリオ**: 平日昼間（高負荷運転中）に毎日アラートが出るようになった

```
月曜日 13:00: アラート発報 → 担当者が確認 → 設備は正常 → 誤検知
火曜日 13:30: アラート発報 → 担当者が確認 → 設備は正常 → 誤検知
水曜日 12:45: アラート発報 → 担当者「どうせ昼間の誤検知だろう」→ 確認をスキップ

木曜日 13:00: アラート発報 → 担当者「また誤検知」→ スキップ
             → 実際には今回は本物の過熱が起きていた
             → 2時間後に機器停止
```

これが「アラート疲れ」の典型的なシナリオ。
誤検知を出し続けると、本物のアラートを見逃す原因になる。

だから閾値設計において「誤検知をどう抑制するか」は「検知漏れをどう防ぐか」と
同じくらい重要な課題。

## 理解度確認（模範解答を読んだ後に答えてみる）

---

**Q1: 温度センサーで「avg+3σ をそのまま使う」と誤検知が発生する理由を説明してください。**

A1: 温度は設備の負荷（電流）に連動して変化するため、高負荷運転中の正常な温度でも
avg+3σ の閾値を超えることがある。振動と違い温度は「時間帯・負荷によって正常値の絶対量が変わる」
センサーなので、絶対値の閾値だけでは不十分になる場合がある。

---

**Q2: 「電流と温度の比率を監視する」アプローチが「最も正確」な理由を説明してください。**

A2: 「電流が高い → 温度が高い」という関係は正常な物理現象。
比率（temperature / current）を監視すれば、負荷の大小に関係なく「電流に対して温度が異常に高い」
状態を検知できる。つまり「負荷変動の影響を受けない」異常検知が可能になる。

---

**Q3: 問題2で `num_events: 5` にした理由を、問題1（`num_events: 3`）と比較して説明してください。**

A3: 温度は振動より誤検知リスクが高い（高負荷時に正常でも閾値付近の値が出やすい）。
そのため条件を厳しくして「短時間に5件以上超える」場合のみアラートを出すようにした。
温度の異常は長時間継続する性質があるため、条件を厳しくしても本物の異常は十分な件数が積み上がる。

---

**Q4: 閾値設計の前に「センサーの特性を調べる」とは具体的に何をすることですか？**

A4: そのセンサーの値が「何に連動して変化するか」を確認すること。
具体的には Kibana で temperature と current を同時に可視化して、
電流が上がったとき温度も上がるかを確認する（相関の確認）。
この確認を省くと「なぜ誤検知が出るのか」が設計後にわからなくなる。

---

**Q5: 「完璧な手法（比率監視）」より「実用的な妥協案（avg+3σ + マージン）」を選ぶ判断はなぜ認められるのですか？**

A5: 完璧な手法には Logstash の改修という実装コストがかかる。
実装コストと得られる精度向上を比較して「今の環境でできる最善の設計」を選ぶことは実務上の正当な判断。
重要なのは「妥協案を選んだ根拠と、今後の改善計画」をドキュメントに残すこと。
