### 「閾値を 8.0 以上にした」場合

**発想の方向性**: 「明らかな異常値だけを拾いたい」 ← 方向性は間違っていない

**問題点**:

まず「予知保全」の目的を振り返る。
「壊れそうになる前に気づく」ことが目的であり、「壊れてから気づく」ことではない。

振動値が 8.0 に達した時点で何が起きているかを考えると:
- 正常値（約 3.2）の2.5倍の振動が出ている
- ベアリングは「すでに著しく劣化した状態」にある
- この時点から部品を発注しても、届く前に故障するかもしれない

つまり 8.0 での検知は「手遅れに気づく」に近い。

**もう一つの問題**: 「なぜ 8.0 なのか」をデータで説明できない。
直感で決めた値は、現場の担当者から「根拠は？」と問われたときに答えられない。

**考え直すポイント**:
閾値の目的は「正常と異常の境界線」を引くこと。
境界線は「正常値が収まる範囲の上限付近」に設定するのが合理的。
正常期間の avg が 3.2 なのに、閾値が 8.0 では「境界線」が遠すぎる。

---

### 「平均値（約 3.2）を閾値にした」場合

**問題点**:

正常期間のデータを想像してほしい。
正常値が平均 3.2 で分布しているということは:
- 3.2 より大きいデータが全体の約 50% ある
- つまり正常なのに閾値を超えるデータが毎日大量に発生する

例えば 1 日に 1440 件のデータがある場合、そのうち約 720 件が閾値を超えてしまう。
これを「誤検知」という。大量の誤検知が出ると、アラートが来るたびに確認作業が発生し、
現場担当者がアラートに慣れてしまって（「どうせ誤検知だろう」）本当の異常を見逃す。
この状態を「アラート疲れ」と呼ぶ。

**考え直すポイント**:
平均値は「正常値の中央」を示す値。
閾値に使う数値は「正常値の上限」付近でなければならない。

---

### 「最大値（約 4.8）をそのまま閾値にした」場合

**問題点**:

正常期間の最大値（4.8）を閾値にすれば「正常期間には一度も閾値を超えない」と思うかもしれない。
しかし現実にはそうならない。

「正常期間の最大値」は「正常期間に一度だけ偶然出た高い値」かもしれない。

たとえばこんなことが起きていた可能性がある:
- ある日だけ工場の稼働率が非常に高く、振動が 4.8 まで上がったが、翌日には戻った
- センサーの計測ノイズで一瞬だけ 4.8 という値が記録された
- メンテナンス作業中に一時的に高い振動が発生した

こういった「偶発的な外れ値」が1件でもあると、閾値が不当に高く設定されてしまう。

**具体例で考える**:

```
正常期間の振動値（イメージ）:
3.0, 3.1, 3.2, 3.3, 3.1, 3.2, 3.0, 4.8, 3.1, 3.2, ...
                                      ↑
                              この1件だけ偶発的に高い
```

この 4.8 は「ノイズ」かもしれない。
最大値をそのまま使うと、このノイズに引きずられて閾値が高くなりすぎる。

**改善案**:
- 最大値を基準にするなら「95パーセンタイル」（上位5%を除いた最大値）の方が安定する
- さらに良いのは avg+3σ を使う方法（後述）

---

### 「変化量（差分）ベースを選んだ」場合

**発想は優秀！** 実際の予知保全の現場でも有効なアプローチ。

ただし今回の環境と今回のデータにはいくつかの課題がある:

**ElastAlert2 での実装課題**:
ElastAlert2 はルールの `filter` で使えるのは「現時点の各フィールドの値」だけ。
「現在値 - 24時間前の値」のような「フィールド間の差分計算」は ElastAlert2 単体ではできない。
実装するには Logstash パイプラインで差分フィールドを事前計算して保存する必要があり、
実装コストが高い。

**今回のデータの特性**:
pump-01 の振動は「ゆっくりと上昇するパターン」をとる。
変化量ベースでは「1時間あたりの増加量が小さい」うちはアラートが出ない。
avg+3σ の方が「累積して上昇した値」に対して早く反応できる。

**この発想の活かし方**:
発展課題として Logstash で差分フィールドを計算するパイプラインを実装すれば、
avg+3σ と組み合わせた多層的なアラート設計が可能になる。

## 統計の基本概念: 標準偏差（σ）とは何か

初めて標準偏差を使う人向けに、数式を使わずに説明する。

### 「ばらつき」を数値化したもの

センサーのデータは毎回まったく同じ値にはならない。
正常なのに少し高い値が出たり、少し低い値が出たりする。
この「正常範囲内での値の揺れ幅」を数値で表したものが標準偏差（σ）。

```
例: pump-01 の正常期間の振動値（イメージ）

    3.0, 3.1, 3.2, 3.3, 3.2, 3.1, 3.0, 3.4, 3.1, 3.2, ...

    平均: 3.2
    ばらつき: ±0.4 程度（多くの値が 3.2 から 0.4 以内に収まっている）
    標準偏差 σ ≈ 0.4
```

### 「正常値の 99.7% が収まる範囲」

正規分布（ベル型の分布）に近いデータでは、以下の法則が成り立つ:

| 範囲 | 含まれるデータの割合 |
|---|---|
| avg ± 1σ（3.2 ± 0.4 = 2.8〜3.6） | 約 68% |
| avg ± 2σ（3.2 ± 0.8 = 2.4〜4.0） | 約 95% |
| avg ± 3σ（3.2 ± 1.2 = 2.0〜4.4） | 約 99.7% |

つまり「avg + 3σ = 4.4」という閾値は:
- 正常時は 99.7% のデータがこの値を下回る
- この値を超えるのは正常時でも 1000 件に 3 件程度（0.3%）

### なぜ「avg + 3σ」が「統計的に意味がある」か

「4.4 以上は異常」という判断の根拠が明確になるから。

```
根拠の示し方:
「正常期間（2026-01-01〜01-12）のデータから計算した統計値 avg=3.2, std=0.4 に基づき、
 正常値の 99.7% が収まる上限（avg + 3σ = 4.4）を閾値として設定した」
```

この説明は誰でも検証できる。
正常期間のデータを再計算すれば同じ 4.4 という値が出る。

「なぜ 8.0 にしたか」「なぜ 6.0 にしたか」は直感で決めた場合は説明できないが、
「なぜ 4.4 にしたか」は統計から算出したので説明できる。

## 誤検知と検知漏れ: 2種類のミスを理解する

アラート設計では必ず 2 種類のミスが存在する。

### 誤検知（False Positive）: 正常なのにアラートが出る

**具体例**:
```
正常稼働中（振動 = 3.1〜3.5 程度）
  ↓
瞬間的に 4.5 という値が1件記録される（ノイズ、測定誤差）
  ↓
閾値 4.4 を超えたのでアラートが発報
  ↓
担当者が確認しに行ったが、設備は正常だった（→ムダな確認作業）
```

誤検知が多いと:
- 担当者がアラートを信じなくなる（「どうせ誤検知だろう」という心理）
- 本当に異常が起きたときにも見逃す可能性が高まる
- 確認作業の工数が無駄に発生する

### 検知漏れ（False Negative）: 異常なのにアラートが出ない

**具体例**:
```
ベアリング劣化が進行中（振動が徐々に上昇）
  ↓
振動値が 4.6 まで上昇（閾値を超えている）
  ↓
しかし閾値を 5.0 に設定していたため、アラートが出ない
  ↓
劣化に気づかずそのまま運転 → 突然の故障
```

検知漏れが多いと:
- 予知保全の効果がなくなる
- 突然故障で生産ラインが止まる
- 最悪の場合、2次被害（他の部品への損傷）が発生する

### トレードオフ: どちらを優先するか

```
閾値を下げる ────────────────────── 閾値を上げる

早期検知できる               検知が遅くなる / 検知漏れのリスク
誤検知が増える               誤検知が減る
「アラート疲れ」のリスク     「手遅れ」のリスク
```

どちらを優先するかは「現場の運用体制」によって変わる。
ただし「根拠なく決めた閾値」は、どちらに設定しても「なぜこの値か」を説明できないため、
現場の信頼を得にくい。

## 4つのアプローチとその評価

### アプローチ A: 固定閾値（直感ベース）

**例**: `vibration > 8.0`

| 指標 | 結果 |
|---|---|
| 最初の検知 | 故障の約 3 日前 |
| 誤検知（正常期間） | ほぼ 0 件 |

**なぜ不十分か**:
予知保全の目的は「壊れる前に気づく」こと。
8.0 という値は「すでに明らかな異常値」であり、この段階では対処が間に合わないことが多い。
また「なぜ 8.0 なのか」をデータで説明できない。

---

### アプローチ B: 正常期間の最大値 × マージン

**例**: `vibration > 4.8 × 1.2 ≈ 5.8`

| 指標 | 結果 |
|---|---|
| 最初の検知 | 故障の約 5〜6 日前 |
| 誤検知（正常期間） | 数件発生する可能性あり |

**評価**: 固定閾値より早く検知できる。
ただしマージン（1.2倍）の根拠が曖昧であり、また正常期間に偶発的に高い値があると
閾値が不当に高く設定されてしまうリスクがある。

---

### アプローチ C: 平均 + 3σ ★ 最も推奨

**例**: `vibration > 3.2 + 3 × 0.4 = 4.4`

| 指標 | 結果（瞬間値で判定） |
|---|---|
| 最初の検知 | 故障の約 6〜7 日前 |
| 誤検知（正常期間） | 数件発生 |

**なぜ推奨か**:
- 正規分布に従うデータでは、平均 ± 3σ の範囲に 99.7% のデータが収まる
- 「なぜ 4.4 か」を「正常期間の統計から算出した」と説明できる
- 設備が変わっても「正常期間の統計を計算する」という同じロジックを使い回せる

**誤検知対策**: `frequency` ルールを組み合わせる

```yaml
type: frequency
num_events: 3       # 15分間に 3 件以上ヒットしたらアラート
timeframe:
  minutes: 15
filter:
  - term:
      device_id.keyword: "pump-01"
  - range:
      vibration:
        gte: 4.4
```

| 指標 | frequency 追加後 |
|---|---|
| 最初の検知 | 故障の約 6 日前（ほぼ変わらず） |
| 誤検知（正常期間） | ほぼ 0 件 |

---

### アプローチ D: 変化量（差分）ベース

**例**: `現在の振動値 - 24時間前の振動値 > 1.5`

**評価**: 発想は有効だが、ElastAlert2 単体ではフィールド間の差分計算ができない。
Logstash で差分フィールドを事前計算する必要があり、実装コストが高い。
また今回のように「ゆっくり上昇する」パターンでは、変化量が小さいため avg+3σ より検知が遅くなりやすい。

## `type: frequency` の詳細解説

ElastAlert2 には複数のルールタイプがある。
なぜここで `type: frequency` を使うのかを丁寧に説明する。

### `type: any` との違い

**`type: any`**:
「条件を満たすデータが 1 件でも見つかったらアラートを発報する」

```yaml
type: any
filter:
  - range:
      vibration:
        gte: 4.4
```

→ 振動が一瞬でも 4.4 を超えたらすぐにアラートが出る

**問題点**:
センサーの計測値には微小なノイズが必ず含まれる。
正常運転中でも「たまたま 4.5 という値が記録される」ことがある。
`type: any` ではこの1件のノイズに反応してアラートが出てしまう。

**`type: frequency`**:
「指定した時間枠（timeframe）の中に、条件を満たすデータが指定件数（num_events）以上あったらアラートを発報する」

```yaml
type: frequency
num_events: 3
timeframe:
  minutes: 15
filter:
  - range:
      vibration:
        gte: 4.4
```

→ 15分間で 3 件以上が 4.4 を超えた場合にのみアラートが出る

### `num_events` と `timeframe` の意味を具体例で理解する

センサーデータが 10 分間隔で記録されているとする（1時間に 6 件）。

**ノイズの場合（正常運転中の偶発的な高い値）**:
```
時刻    振動値   4.4超えか
14:00   3.1      ×
14:10   3.2      ×
14:20   4.6      ○ ← たまたま高い値
14:30   3.1      ×
14:40   3.0      ×
14:50   3.2      ×
```
15分間（14:20〜14:35）で条件を満たしたのは 1 件のみ → 3 件に届かないのでアラートなし

**本物の異常の場合（振動が継続して上昇している）**:
```
時刻    振動値   4.4超えか
14:00   4.5      ○
14:10   4.6      ○
14:20   4.8      ○ ← 15分間で 3 件 → アラート！
14:30   5.0      ○
14:40   5.1      ○
```
15分間（14:00〜14:15）で 3 件以上 → アラートが発報される

### `num_events` をいくつにすべきか

データの記録間隔と、どの程度の「継続性」を求めるかによる。

10分間隔のデータで `timeframe: minutes: 15` の場合:
- 15分間に最大 2 件（00分と10分の 2 件）が入る
- `num_events: 3` にすると「実質的に 20〜30 分連続して超えた場合」が条件になる

5分間隔のデータの場合は `num_events: 3` で「15分間に3件」= 15分間継続が条件になる。

今回のデータは 10 分間隔なので、`num_events: 3, timeframe: minutes: 30` が
「30 分間に 3 件」= 「30 分間の継続的な異常」という意味になる。
誤検知と検知タイミングのバランスを見ながら調整すること。

**最終設計で `timeframe: minutes: 15` を採用した理由**:

今回のデータは 10 分間隔だが、ElastAlert2 の実際の動作では
検索ウィンドウとデータ到達タイミングのずれにより、
`timeframe: 15分` でも 2〜3 件が同じウィンドウに入ることがある。
また「15 分という短い時間枠で 3 件」は「ほぼ連続した計測で閾値を超え続けている」ことを意味し、
検知の即時性と誤検知抑制のバランスが良い。

`timeframe: 30分` も有効な選択肢であり、より厳格に「30 分継続」を確認したい場合はそちらを使う。
今回は「検知タイミングをなるべく早くしたい」という観点で 15 分を採用した。

## 模範解答の最終設計

```yaml
name: "pump-01 振動異常（統計的閾値 + frequency）"
type: frequency
index: sensor-data-*
num_events: 3
timeframe:
  minutes: 15
filter:
  - term:
      device_id.keyword: "pump-01"
  - range:
      vibration:
        gte: 4.4    # avg(3.2) + 3σ(1.2) = 4.4
realert:
  minutes: 60
alert:
  - debug
```

**閾値の根拠（ドキュメントとして残す内容）**:

```
設備: pump-01
センサー: 振動（vibration）
正常期間: 2026-01-01〜2026-01-12（故障履歴を確認、この期間に故障なし）
算出方法: 正常期間の avg（3.2）+ 3σ（0.4 × 3 = 1.2）= 4.4
根拠: 正規分布に基づく 99.7% 信頼区間の上限値
frequency 設定: 15分間に3件以上 → ノイズによる誤検知を抑制
設定日: 2026-01-XX  設定者: ○○
```

このような根拠をコメントや設計書として残すことが、現場運用で非常に重要になる。

## 「誤検知 vs 検知タイミング」のトレードオフ

```
閾値を下げる ─────────────────────→ 閾値を上げる

早期検知できる                       検知が遅くなる
誤検知が増える                       誤検知が減る
「アラート疲れ」のリスク             「手遅れ」のリスク
```

どちらを許容するかは「現場の運用体制」によって変わる。
ただし「根拠なく決めた閾値」は、どちらに設定しても「なぜこの値か」を説明できないため、
現場の信頼を得にくい。

## 「平均 vs 最大値」の答え

閾値の基準値として使う場合:
- **最大値だけ**: 偶発的な高い値があると閾値が不当に高くなる
- **平均値だけ**: 正常値の中央値なので、すぐ誤検知する
- **avg + Nσ**: 「正常値の広がり（ばらつき）」を考慮した統計的な閾値 → 最もバランスが良い

判定に使う値（ElastAlert2 の filter）:
- **瞬間値**: ノイズに敏感、誤検知しやすい
- **frequency（複数件）**: 継続的な変化のみに反応できる → 推奨

---

## 「根拠をドキュメントに残す」ことの重要性

予知保全システムを構築した後で必ず起きることがある。

**半年後に起きること（実際の現場で頻繁に起きる）**:

```
上司や現場担当者: 「最近このアラートが頻繁に出るが、なぜ 4.4 という数値にしたのか？」

直感ベースで設定した場合:
  → 「なんとなく…」「当時の感覚で…」
  → 根拠を説明できない → 信頼を失う

統計ベースで設定し、ドキュメントに残した場合:
  → 「正常期間（2026-01-01〜01-12）のデータから算出した avg+3σ の値です。
     当時の統計: avg=3.2, std=0.4 → 閾値=4.4」
  → 根拠が明確 → 「では正常期間のデータが変わったのか確認しよう」という建設的な議論ができる
```

また閾値の見直しをするときにも、「なぜそうなったか」の経緯がわかっていると
改善点を見つけやすくなる。

**設計書に残す最低限の項目**:
1. 正常期間はいつからいつか（故障履歴のどのデータを参照したか）
2. 正常期間の統計値（avg, std）
3. 閾値の計算式（avg + 3σ など）
4. frequency の設定と、その理由
5. 設定日と設定者

## 理解度確認（模範解答を読んだ後に答えてみる）

模範解答を一通り読んだら、以下の質問に自分の言葉で答えてみること。
答えを見る前に自分で考えることで定着する。

---

**Q1: 「avg+3σ」で閾値を設定する最大の利点を1文で説明してください。**

A1: 正常期間のデータから統計的に算出するため、「なぜこの値か」を数値で根拠を示せ、
誰でも同じデータから同じ値を再現できるから。

---

**Q2: 閾値を低くすると何が増え、高くすると何が増えますか？**

A2: 低くすると誤検知（正常なのにアラートが出る）が増え、
高くすると検知漏れ（または検知が遅くなる）が増える。
どちらも現場にとって問題であり、バランスをとることが重要。

---

**Q3: `type: any` と `type: frequency` の違いは何ですか？また、なぜ `frequency` の方が良いのですか？**

A3: `type: any` は条件を1回でも満たしたらアラート。
`type: frequency` は指定した時間内に指定件数以上ヒットしたらアラート。
センサーには測定ノイズが含まれるため、1件のみでは誤検知しやすい。
`frequency` を使うことで「継続的な異常」にのみ反応できるため誤検知を抑制できる。

---

**Q4: 閾値の「根拠をドキュメントに残す」ことがなぜ重要ですか？**

A4: 半年後に「なぜこの閾値にしたのか」を問われたとき、説明できないと現場の信頼を失う。
また、再設計するときに見直しの基準になるため、過去の判断経緯がわかっていると改善しやすい。

---

**Q5: 正常期間の「最大値（約 4.8）」をそのまま閾値にした場合、どんな問題が起きますか？**

A5: 最大値は「正常期間に偶発的に発生した高い値（ノイズ・一時的な負荷増大など）」
かもしれないため、閾値が不当に高く設定されてしまう可能性がある。
結果として、本当の異常が発生しても閾値に達せず、検知が遅れるリスクがある。

---

**Q6: 「標準偏差（σ）」を数式なしで説明してください。**

A6: 正常なデータが「平均値からどれくらいばらついているか」を表す数値。
σ が小さいと「値が平均値の近くに集まっている（ばらつきが小さい）」ことを意味し、
σ が大きいと「値が広い範囲に散らばっている（ばらつきが大きい）」ことを意味する。
