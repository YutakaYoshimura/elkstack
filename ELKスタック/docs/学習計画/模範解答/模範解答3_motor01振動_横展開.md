### 「pump-01 の閾値（4.4）をそのまま motor-01 のルールにコピーした」場合

**最も多い回答パターン**。その気持ちはよくわかる。

「問題1でやった通りに作ればいいんでしょ？ device_id を変えて、閾値はさっきと同じ 4.4 で。」

**問題点**:
テストを実行したとき、どちらかの問題が起きなかったか。

- 正常期間にアラートが大量に出た → motor-01 の正常振動値が pump-01 より高い
- 故障前後でもアラートが一切出なかった → motor-01 の異常振動値が pump-01 の閾値より低い

どちらの方向にずれても「機能しないルール」になる。

**気づいてほしいこと**:
pump-01 と motor-01 は「設備の種類が違う（ポンプとモーター）」。
設備が違えば、正常稼働時の振動の大きさも、振動のばらつき方も異なる。
閾値の「数値 4.4」は pump-01 というこの設備のこの期間のデータから生まれた値であり、
他の設備に適用できる普遍的な値ではない。

## 問題3の核心: 何が「流用できて」何が「流用できないか」

pump-01 の閾値（数値）をそのまま motor-01 に適用すると、
多くの場合で以下のどちらかが起きる:

**ケース1: motor-01 の正常振動値が pump-01 より高い場合**

```
pump-01 の avg+3σ ≈ 4.4 を motor-01 に適用
motor-01 の正常最大値が 5.5 だとすると
→ 正常な振動値（5.0 など）でもアラートが発生する（大量の誤検知）
```

**ケース2: motor-01 の正常振動値が pump-01 より低い場合**

```
pump-01 の avg+3σ ≈ 4.4 を motor-01 に適用
motor-01 の正常最大値が 3.0 だとすると
→ 故障前兆の振動値が 3.5 程度でも、閾値 4.4 に達しない（検知できない）
```

どちらの方向にずれても、「pump-01 の閾値値をそのまま使う」は機能しない。

### 数値で確認する

pump-01 と motor-01 の正常期間統計を並べると、その違いが明確になる:

| 設備 | avg | std | avg+3σ（閾値） |
|---|---|---|---|
| pump-01 | 約 3.2 | 約 0.4 | 約 4.4 |
| motor-01 | 各自計算 | 各自計算 | 各自計算（pump-01 とは異なる値） |

motor-01 の統計値は自分の API 実行結果で確認すること。
pump-01 の値と比較して「なぜ違うのか」を考えることが学びになる。

## 正しいアプローチ: 設備ごとに正常統計から計算する

motor-01 専用の閾値を求めるために、motor-01 の正常期間の統計を計算する。

**motor-01 の正常期間（2026-01-01〜2026-01-25）の統計（参考値）**:

| 統計量 | 参考値（pump-01 との比較） |
|---|---|
| 平均 (avg) | 約 ○○（pump-01 は約 3.2） |
| 最大値 (max) | 約 ○○（pump-01 は約 4.8） |
| 標準偏差 (std) | 約 ○○（pump-01 は約 0.4） |
| avg + 3σ | 約 ○○（pump-01 は約 4.4） |

> 実際の値は各自の API 実行結果で確認すること。

motor-01 の avg+3σ が pump-01 と同じになることは、まずない。
設備タイプ（ポンプとモーター）が違えば、振動の特性も異なる。

## 「ロジックのコピー」と「数値のコピー」の違いを理解する

これが問題3で最も重要な概念。

```
正しい横展開（ロジックのコピー）:

pump-01 でやったこと:
  1. 正常期間を特定する（2026-01-01〜01-12）
  2. 正常期間の avg と std を計算する（avg=3.2, std=0.4）
  3. avg+3σ を閾値にする（4.4）
  4. frequency ルールを設定する

motor-01 でやること:
  1. 正常期間を特定する（2026-01-01〜01-25）  ← 手順は同じ
  2. 正常期間の avg と std を計算する（○○, ○○）← 手順は同じ
  3. avg+3σ を閾値にする（○○）               ← 手順は同じ、数値は違う
  4. frequency ルールを設定する               ← 手順は同じ

→ 手順（ロジック）は完全に同じ
→ 数値だけが設備ごとに異なる
```

これを「設計ロジックの再利用」と呼ぶ。

「数値のコピー」との違い:
```
誤った横展開（数値のコピー）:

pump-01 の結果:  avg=3.2, std=0.4, 閾値=4.4
motor-01 に適用: 閾値=4.4（pump-01 の計算結果をそのまま転用）

→ motor-01 の正常期間のデータを見ていないため、
  motor-01 の「正常な振動の大きさ」を全く考慮していない
→ 動かないか、誤検知/検知漏れのどちらかが起きる
```

## 模範解答の設計

```yaml
name: "motor-01 振動異常（motor-01専用の統計から設計）"
type: frequency
index: sensor-data-*
num_events: 3
timeframe:
  minutes: 15
filter:
  - term:
      device_id.keyword: "motor-01"
  - range:
      vibration:
        gte: ○○    # motor-01 の avg + 3σ（自分の統計結果から計算）
realert:
  minutes: 60
alert:
  - debug
```

**設計の根拠**:
- ロジックは問題1と同じ（avg+3σ + frequency）
- 閾値の値（数値）は motor-01 の正常期間統計から計算した独自の値

**閾値根拠の記録**:

```
設備: motor-01
センサー: 振動（vibration）
正常期間: 2026-01-01〜2026-01-25（failure_history.csv で故障日を確認し、故障前の安定期間を選定）
算出方法: 正常期間の avg（○○）+ 3σ（○○ × 3 = ○○）= ○○
根拠: pump-01 と同じ設計ロジック（avg+3σ）を適用。数値は motor-01 の独自統計から算出。
設定日: 2026-XX-XX  設定者: ○○
```

## 「流用すべきもの」と「流用してはいけないもの」

```
設備横展開のルール:

○ 流用すべきもの（ロジック・手法）:
  - 「正常期間の統計（avg+3σ）を閾値にする」という設計思想
  - frequency ルールで誤検知を抑制するという構造
  - ElastAlert2 のルールファイルの書き方（YAML の書式）
  - 「正常期間の特定 → 統計計算 → 閾値算出 → テスト」という手順

× 流用してはいけないもの（具体的な数値）:
  - pump-01 で出た「4.4」という閾値の数値そのもの
  → 設備ごとに正常期間のデータから再計算する
```

## 「横展開」の正しい手順

1. **新しい設備の正常期間を特定する** — いつからいつが「故障の影響がない時期」か
2. **その設備の正常期間の統計を計算する** — avg, std を出す
3. **avg+3σ（または適切なアプローチ）で閾値を算出する**
4. **ルールファイルを設備専用に作成する**（device_id と閾値の値を変える）
5. **テストで「誤検知件数」と「検知タイミング」を確認する**

この手順は pump-01 でも motor-01 でも、さらに設備が100台になっても変わらない。
**同じ手順を繰り返せる**ことが、統計ベースのアプローチの強みである。

## なぜ「ロジックを再利用できること」が重要か

### 設備が増えたときのコスト比較

**直感ベースの閾値（固定値）を使っていた場合、設備が 10 台になったとき**:

```
設備が増えるたびに:
  ・「この設備の閾値はいくつが適切か」を毎回直感で考え直す
  ・根拠がないため、誰かが疑問を持つたびに再検討が必要
  ・設備が違えば閾値も違うが、「なぜ違うのか」を説明できない
```

**統計ベースのロジックを使っていた場合、設備が 10 台になったとき**:

```
設備が増えるたびに:
  ・その設備の正常期間データを取得する（手順は同じ）
  ・avg と std を計算する（手順は同じ）
  ・avg+3σ を計算する（手順は同じ）
  ・「なぜこの閾値か」は常に同じ説明でよい
  → 設備が 100 台でも同じフローを繰り返すだけ
  → さらにプログラムで自動化できる
```

「同じロジックを繰り返せる」設計は、スケールする（規模が大きくなっても対応できる）。

### 自動化への布石

統計ベースのロジックは自動化できる:

```python
# イメージ（疑似コード）
for device_id in ["pump-01", "pump-02", "motor-01", ...]:
    stats = calculate_normal_period_stats(device_id)
    threshold = stats["avg"] + 3 * stats["std"]
    create_elastalert_rule(device_id, threshold)
```

「設備ごとに人が閾値を考える」のではなく、
「プログラムが各設備のデータから閾値を自動計算する」ことができる。
これが統計ベースアプローチの最大のメリットの一つ。

## 問題3から得るべき重要な気づき

### 1. 「スケールアウト」の落とし穴

設備が増えたとき、最も楽に見える方法は「閾値をコピーする」こと。
しかしこの問題で体験したように、それは正しく動かない可能性が高い。

**正しい横展開 = 「設計ロジックをコピーする」**

ロジックが同じであれば、たとえ設備が100台になっても
「その設備の正常期間のデータを取り→統計を計算→閾値を算出」というフローを自動化できる。

### 2. 「なぜ avg+3σ が再現性に優れているか」の実感

pump-01 の閾値（数値）は再現できないが、
pump-01 の「設計ロジック（avg+3σ）」はどの設備にも再現できる。

これが「固定閾値（直感ベース）」より「統計ベース」が優れている最大の理由の一つ。

### 3. 設備タイプが違えば、さらに特性も違う

今回は同じ「振動センサー」「ベアリング劣化」という条件でも数値が違った。
設備タイプが全く異なる場合（ポンプとコンプレッサー、振動と圧力など）は、
センサーの特性自体が変わる可能性があり、追加の考慮が必要になることもある。

### 4. 正常期間の「選び方」も設計の一部

「どのデータを正常期間と呼ぶか」を正しく判断することが、閾値精度に直結する。
正常期間に故障前兆のデータが混ざっていると、avg や std が歪み、閾値も不正確になる。

正常期間の選び方の鉄則:
- 故障履歴を確認し、故障日より前の十分な期間を選ぶ
- 季節変動・長期トレンドがある場合は直近の安定期間を選ぶ
- 正常期間のグラフを目視で確認し、明らかな異常値が含まれていないか確認する

## 理解度確認（模範解答を読んだ後に答えてみる）

---

**Q1: pump-01 の閾値（4.4）を motor-01 にそのまま使うと、どんな問題が起きる可能性がありますか？2つ答えてください。**

A1:
(1) motor-01 の正常振動値が pump-01 より高い場合 → 正常なのに大量の誤検知が発生する
(2) motor-01 の正常振動値が pump-01 より低い場合 → 異常が起きても閾値に達せず検知できない（検知漏れ）

---

**Q2: 「ロジックのコピー」と「数値のコピー」の違いを具体的に説明してください。**

A2:
ロジックのコピーとは「avg+3σ を閾値にする」という計算手順をそのまま使うこと。
数値のコピーとは pump-01 の計算結果（4.4 という値）をそのまま motor-01 に当てはめること。
ロジックは再利用できるが、数値は設備ごとのデータから再計算する必要がある。

---

**Q3: 横展開の手順を、自分の言葉で順番に説明してください。**

A3:
1. 新しい設備の故障履歴を確認し、正常期間（故障の影響がない期間）を特定する
2. その期間のデータから avg（平均）と std（標準偏差）を計算する
3. avg+3σ を計算して閾値を出す
4. ElastAlert2 のルールファイルを設備専用に作成する（device_id と閾値を変える）
5. テストで誤検知件数と検知タイミングを確認する

---

**Q4: 統計ベースのアプローチが「自動化しやすい」のはなぜですか？**

A4: 「正常期間の統計を計算して avg+3σ を閾値にする」という手順が設備ごとに同じのため、
プログラムでループ処理できる。直感ベースの閾値設計は「人が考えること」が必要なので
自動化できないが、統計ベースなら各設備のデータを渡せばプログラムが閾値を計算できる。

---

**Q5: 「正常期間の選び方」を間違えると、どんな影響が閾値に出ますか？**

A5: 正常期間に故障前兆（徐々に上昇するデータ）が混ざっていると、
avg が本来の正常値より高くなり、std も大きくなる。
その結果、avg+3σ で計算した閾値が不当に高くなり、
本物の異常が発生しても閾値に達しない（検知漏れ）が起きやすくなる。

---

**Q6: pump-01 と motor-01 で avg+3σ の数値が異なった。それは「想定内」ですか、「計算ミス」ですか？**

A6: 想定内。設備の種類（ポンプとモーター）が違えば、正常稼働時の振動の大きさや
ばらつき方も異なるため、統計値も異なる。
もし pump-01 と motor-01 の avg+3σ がほぼ同じ値になった場合は、
むしろ「計算に誤りがある可能性」や「同じデータを使ってしまった可能性」を疑うべき。
