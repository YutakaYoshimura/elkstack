## Day 1〜2: データの流れを設定ファイルから読む

### sensor-data.conf（Logstash パイプライン）の構造

この設定ファイルは **CSV ファイルのデータを Elasticsearch に取り込むための定義**。

| セクション | 役割 | 今回の設定内容 |
|---|---|---|
| input | どこからデータを読むか | `/data/sensor_data.csv` を先頭から読む |
| filter | どう加工するか | CSV分割、数値型変換（float）、タイムスタンプ変換 |
| output | どこへ送るか | Elasticsearch の `sensor-data-YYYY.MM` インデックスへ |

### 学んだ重要概念

#### sincedb_path

- Logstash が「どこまで読んだか」を記録する仕組み
- 有効にすると再起動時に続きから読める（本番向け・重複防止）
- `/dev/null` に設定すると毎回先頭から全件読み直す（検証環境向け）

#### Elasticsearch と RDB の違い

| RDB の概念 | Elasticsearch の概念 | 今回の例 |
|---|---|---|
| テーブル | インデックス | `sensor-data-2026.01` |
| 行 | ドキュメント | センサーデータ1件 |
| 列 | フィールド | `vibration`, `temperature` 等 |
| スキーマ定義 | マッピング | フィールドの型情報 |

#### 動的マッピング

- RDB のような事前の `CREATE TABLE` は不要
- 最初のデータ投入時にインデックスとマッピングが **自動生成** される
- ただし **一度作られたフィールドの型は変更できない**（変更には reindex が必要）

#### フィールド追加時の挙動

- 新しいフィールドを持つデータを送れば自動で追加される
- 既存ドキュメントには追加されない（過去データは `null` 扱い）
- フィールドが存在しないドキュメントは **エラーにならず無視される**
- 設計思想: 「あるものだけ使う、ないものは無視する」

#### text 型と keyword 型

| 型 | 保存方法 | 用途 |
|---|---|---|
| `text` | 文字列を分解（"ポンプA-01" → "ポンプ", "A", "01"） | 全文検索 |
| `keyword` | 文字列をそのまま保存 | 完全一致検索・ソート・集計 |

- KQL で `.keyword` を付けるのは keyword 型で完全一致検索するため
- `ignore_above: 256` は 256文字超の文字列を keyword にインデックスしない設定（デフォルト値）

#### mutate convert（型変換）の重要性

- これをしないと数値が文字列として保存される
- 文字列のままでは範囲検索や集計ができない
- **最初の投入時に正しい型にすることが重要**（後から型変更はできないため）

## Day 3〜4: 正常期間の統計分析

### Kibana Discover での目視確認

- pump-01 の正常期間（1/1〜1/12）を KQL で絞り込み
- 振動値は **2.5〜3.5 の範囲** に収まっていることを目視確認

### extended_stats API の結果

| 指標 | 正常期間（1/1〜1/12） | 故障前後（1/13〜1/20） |
|---|---|---|
| count | 3,456件 | 2,304件 |
| min | 1.85 | 1.97 |
| max | 4.18 | **6.52** |
| avg | **3.00** | **3.88** |
| std | **0.306** | **0.964** |

### 統計的な閾値の計算

```
avg + 2σ = 3.00 + 2 × 0.306 = 3.61
avg + 3σ = 3.00 + 3 × 0.306 = 3.92
```

### 仮の閾値の決定

- **選んだ閾値: 3.92（avg + 3σ）**
- 理由: 異常予測は大事だが誤検知が多すぎるのも問題。99.7% の精度はバランスが良い

## Day 5〜7: ダッシュボード作成

### 作成したダッシュボード: 「振動センサー監視（pump-01）」

4つのパネルを作成:

| パネル | 種類 | 内容 |
|---|---|---|
| 1 | 折れ線グラフ | pump-01 の振動値時系列 + 閾値ライン（3.92） |
| 2 | Metric | pump-01 の最大振動値（6.52） |
| 3 | Datatable | 全設備の振動統計（平均・最大・最小） |
| 4 | 折れ線グラフ | 全設備の振動推移（色分け比較） |

### ダッシュボードから読み取れたこと

- pump-01 の故障前兆は **1/13頃** から始まり、**1/19〜20** でピーク
- 閾値 3.92 は **正常期間の誤検知なし**、**故障の約4〜5日前に検知可能**
- 3台の設備は正常時の振動基準値が異なる（pump-01: 3.0, pump-02: 2.8, motor-01: 4.0）

### Kibana 操作メモ

- 色分け（Break down by）は日本語UIでは **「内訳」** セクション
- 参照線（Reference line）は **「レイヤーを追加」→「基準線」** で設定

## Week 1 で学んだことの全体像

```
CSV → Logstash → Elasticsearch → Kibana
       (加工)     (保存・検索)    (可視化)

1. データの流れは一方向（CSV → ES）。Kibana は ES を読み取るだけ
2. Elasticsearch は動的マッピングで柔軟にデータを受け入れる
3. ただし型変更はできないので、最初の投入設計が重要
4. 統計（avg, std）に基づいて閾値を決めることで、根拠のある異常検知が可能
```
