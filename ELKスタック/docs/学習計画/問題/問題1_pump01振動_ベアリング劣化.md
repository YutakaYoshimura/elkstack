## この問題で学ぶこと

- Elasticsearch の統計 API（extended_stats）を使ってデータの特性を把握する
- 統計値（平均・標準偏差）をもとに「科学的な閾値」を設計する
- ElastAlert2 のルールファイル（YAML）を自分で書く
- テスト実行結果を読み解き、閾値の良し悪しを評価する

## 問題の背景

pump-01 は 2026-01-20 にベアリング劣化で停止した。
振動センサー（`vibration` フィールド）を使って、**この故障を事前に検知する閾値を設計する**。

「事前に検知」とは、故障日（1/20）よりも前にアラートが出ることを意味する。
検知が早ければ早いほど、保守員が対応できる時間が増えて良い。
ただし「検知が早すぎる = 閾値が低すぎる」と正常時でもアラートが出てしまう（誤検知）。

**閾値設計のゴール**: 誤検知をなるべく少なくしながら、故障前にアラートを出す

**使うデータ**:
- 設備: `pump-01`
- センサー: `vibration`
- 故障日: 2026-01-20
- 正常期間（閾値の基準にする）: 2026-01-01〜2026-01-12

## Step 1: 正常期間の統計を自分で調べる

まず「正常なときの振動値はどのくらいか」を数字で把握する。
勘や目視ではなく、**API で統計値を計算**して確認する。

### 統計 API を実行する

以下のコマンドを PowerShell で実行する。

```powershell
curl.exe -u elastic:changeme -X POST `
  "http://localhost:9200/sensor-data-2026.01/_search?pretty" `
  -H "Content-Type: application/json" `
  -d '{
    \"size\": 0,
    \"query\": {
      \"bool\": {
        \"filter\": [
          {\"term\": {\"device_id.keyword\": \"pump-01\"}},
          {\"range\": {\"@timestamp\": {\"gte\": \"2026-01-01\", \"lt\": \"2026-01-13\"}}}
        ]
      }
    },
    \"aggs\": {
      \"vibration_stats\": {
        \"extended_stats\": {\"field\": \"vibration\"}
      }
    }
  }'
```

---

> ### コラム: このコマンドは何をしているのか
>
> `curl` は HTTP リクエストを送るコマンドラインツール。
> Elasticsearch は REST API を提供しており、HTTP でデータの検索・集計ができる。
>
> このコマンドを日本語に訳すと：
> 「`sensor-data-2026.01` というインデックス（データベースのテーブルに相当）に対して、
> device_id が pump-01 で、2026-01-01〜01-12 のデータを絞り込み、
> vibration フィールドの統計（extended_stats）を計算せよ」
>
> オプションの意味：
> - `-u elastic:changeme` : ユーザー名とパスワード（認証）
> - `-X POST` : HTTP メソッドを POST に指定
> - `-H "Content-Type: application/json"` : 送るデータが JSON 形式であることを宣言
> - `-d '...'` : 送る JSON の本文

### API 実行結果の読み方

実行すると、以下のような JSON が返ってくる。各フィールドの意味を理解してから、自分の結果と照らし合わせること。

```json
{
  "aggregations": {
    "vibration_stats": {
      "count": 1728,
      "min": 2.45,
      "max": 4.83,
      "avg": 3.21,
      "sum": 5546.88,
      "sum_of_squares": 18234.5,
      "variance": 0.1444,
      "std_deviation": 0.38,
      "std_deviation_bounds": {
        "upper": 3.97,
        "lower": 2.45,
        "upper_population": 3.97,
        "lower_population": 2.45
      }
    }
  }
}
```

| フィールド名 | 意味 | 使い道 |
|---|---|---|
| `count` | データ件数 | 十分なデータがあるか確認（少ないと統計が不安定） |
| `min` | 最小値 | 正常時の下限 |
| `max` | 最大値 | 正常時の上限（この値を大幅に超えたら異常の可能性） |
| `avg` | 平均値 | 正常時の「典型的な値」 |
| `std_deviation` | 標準偏差（σ） | 値のばらつき具合（詳細は後述） |
| `std_deviation_bounds.upper` | 平均 + 2σ | 95.4% のデータがこの値以下 |
| `std_deviation_bounds.lower` | 平均 - 2σ | 95.4% のデータがこの値以上 |

---

> ### コラム: 標準偏差（σ）とは何か
>
> 標準偏差は「正常値がどのくらいばらついているか」を表す数値。
>
> 具体例で考えてみよう：
> - センサーA の振動値が「3.0, 3.1, 2.9, 3.0, 3.1, 2.9 ...」と安定している → std が小さい（例: 0.1）
> - センサーB の振動値が「2.0, 4.0, 1.5, 4.5, 2.5, 3.5 ...」と激しく変動する → std が大きい（例: 1.0）
>
> **std が小さいセンサー（安定）**: 少し値が上がっただけで「異常かも」と気づきやすい
> **std が大きいセンサー（不安定）**: 閾値を高めに設定しないと誤検知が多発する
>
> ---
>
> **平均 + 3σ（avg + 3σ）の意味:**
>
> 正規分布（釣り鐘型の分布）に従うデータでは：
> - 平均 ± 1σ の範囲に **68.3%** のデータが収まる
> - 平均 ± 2σ の範囲に **95.4%** のデータが収まる
> - 平均 ± 3σ の範囲に **99.7%** のデータが収まる
>
> つまり「avg + 3σ を超える値」は **1000件に3件しか起きない、滅多にない値**。
> これを閾値にすると、「正常時でもたまたま閾値を超える確率」を 0.3% 以下にできる。
>
> 図で表すと：
> ```
> ←---- 平均-3σ ----← 平均-2σ ←-- 平均-σ ← 平均 → 平均+σ --→ 平均+2σ ----→ 平均+3σ --→
>
>       |←────────────────── 99.7% のデータがここに収まる ──────────────────→|
>                   |←────────── 95.4% のデータがここに収まる ──────────→|
>                            |←── 68.3% のデータがここに収まる ──→|
>
> ここより右は稀（0.15%）↑                         ↑ ここより右は稀（0.15%）
> （異常の可能性）                                   （異常の可能性）
> ```
>
> 振動センサーの場合、「下に外れる（小さすぎる）」はあまり問題にならないことが多いので、
> 上限の `avg + 3σ` だけを閾値として使う。

### 「瞬間値（type: any）」と「複数件判定（type: frequency）」の違い

閾値設計では、**何をトリガーにするか** もポイント。

| 方法 | ElastAlert2 の設定 | 内容 | 特徴 |
|---|---|---|---|
| 瞬間値 | `type: any` | 1 回でも閾値を超えたらアラート | ノイズに敏感。1 件の突発値でも反応する |
| 複数件判定 | `type: frequency` | 指定時間内に N 件以上閾値を超えたらアラート | ノイズに強い。継続的な変化のみに反応する |

> **注意**: 「複数件判定」は「直近 N 回の平均を計算して閾値と比べる」わけではない。
> 「指定した時間枠の中に条件を満たすデータが N 件以上あれば」という判定方法（カウント）。
> 移動平均を使いたい場合は Logstash で事前に計算フィールドを追加する必要がある。

ベアリング劣化は「ある日突然ガクンと悪化する」というより、「じわじわと悪化する」傾向がある。
どちらの方が適しているか、理由とともに考えてみよう。

## Step 3: 実装してテストする

Step 2 で選んだ案を ElastAlert2 のルールファイルとして実装する。

ファイルのパス: `elastalert/rules/vibration-threshold-my.yaml`

### YAML ファイルを作成する前に（重要）

---

> ### コラム: ElastAlert2 の YAML ルールファイルとは
>
> ElastAlert2 は「ルールファイル（YAML 形式）」を読み込んで動作する。
> ルールファイルには「どのデータを、どんな条件で監視し、条件を満たしたら何をするか」を記述する。
>
> **YAML の基本ルール（これを守らないとエラーになる）:**
>
> 1. **インデントはスペース2つ**（タブ文字は使わない。エディタによってはタブが自動挿入されるので注意）
> 2. **コロン（:）の後にはスペースが必要**
>    - 正しい: `name: "テスト"`
>    - 間違い: `name:"テスト"`（スペースなし）
> 3. **文字列はダブルクォートで囲む**（数値は不要）
>    - 文字列: `name: "pump-01 アラート"`
>    - 数値: `minutes: 60`
> 4. **ハイフン（-）でリスト要素を表す**
>    - `filter:` の下の `- term:` は「フィルター条件の1つ目」を意味する
>
> **各フィールドの意味:**
>
> ```yaml
> name: "ルール名"         # このルールの名前（ログに表示される）
> type: any                # ルールタイプ。any = 条件を満たすデータが1件でもあればアラート
> index: sensor-data-*     # 監視するインデックス。* はワイルドカード（全月を対象）
>
> filter:                  # 絞り込み条件（AND 条件）
>   - term:                # term = 完全一致検索
>       device_id.keyword: "pump-01"   # device_id が pump-01 のデータのみ
>   - range:               # range = 数値範囲で絞り込み
>       vibration:
>         gte: 5.0         # gte = Greater Than or Equal（5.0 以上）
>                          # gt = Greater Than（5.0 より大きい）
>                          # lte = Less Than or Equal（以下）
>                          # lt = Less Than（未満）
>
> realert:
>   minutes: 60            # 同じルールのアラートを60分間は再送しない（通知の氾濫を防ぐ）
>
> alert:
>   - debug                # debug = コンソールにログを出力するだけ（メールは送らない）
>                          # テスト中はこれを使う
> ```
>
> **device_id.keyword とは？**
> Elasticsearch では文字列フィールドに `text` と `keyword` の2種類がある。
> `term`（完全一致）で検索するときは `.keyword` を使う必要がある。
> `device_id.keyword: "pump-01"` と書かないと正しく絞り込めない。

---

### YAML ファイルを作成する

以下の内容で `elastalert/rules/vibration-threshold-my.yaml` を作成する。
`gte: ○○` の部分を、Step 2 で自分が決めた値に書き換えること。

```yaml
name: "pump-01 振動閾値アラート（自分の設計）"
type: any
index: sensor-data-*

filter:
  - term:
      device_id.keyword: "pump-01"
  - range:
      vibration:
        gte: 5.0    # ← ここを自分が決めた値に変更する

realert:
  minutes: 60

alert:
  - debug
```

---

> ### コラム: docker exec の意味
>
> テストコマンドに登場する `docker exec` について説明する。
>
> ```powershell
> docker exec elastalert elastalert-test-rule ...
> ```
>
> これは「**elastalert というコンテナの中で elastalert-test-rule コマンドを実行せよ**」という意味。
>
> **コンテナとは**:
> コンテナとは、アプリケーションが動作する「隔離された仮想環境」のこと。
> ElastAlert2 は Windows の PC 上で直接動いているのではなく、Docker が作ったコンテナの中で動いている。
> そのため、ElastAlert2 のコマンドを実行するには「コンテナの中に入って実行する」必要がある。
>
> `docker exec [コンテナ名] [コマンド]` が基本構文。
>
> **ルールファイルのパスについて**:
> ローカルのパスは `elastalert/rules/vibration-threshold-my.yaml` だが、
> コマンドでは `/opt/elastalert/rules/vibration-threshold-my.yaml` と書く。
> これは `docker-compose.yml` でローカルの `elastalert/rules/` ディレクトリが
> コンテナ内の `/opt/elastalert/rules/` にマウント（共有）されているため。

| 確認項目 | 良い結果の目安 | 注意が必要な場合 |

| 最初のアラート日時 | 故障日の数日〜1週間前 | 故障日当日や翌日なら閾値が高すぎる |
| 故障前期間のヒット数 | ある程度の件数 | 0件なら閾値が高すぎて検知できていない |

> ### コラム: エラーが出た場合の対処法
>
> **よくあるエラーとその原因:**
>
> `yaml.scanner.ScannerError: mapping values are not allowed here`
> → YAML のインデントかコロンの書き方が間違っている。コロンの後にスペースがあるか確認する。
>
> `FileNotFoundError: /opt/elastalert/rules/vibration-threshold-my.yaml`
> → ファイル名やパスが間違っている。ファイルが `elastalert/rules/` に保存されているか確認する。
>
> `ConnectionError: Failed to connect to Elasticsearch`
> → Elasticsearch が起動していない可能性がある。
> `docker compose ps` で状態を確認し、`docker compose up -d` で起動する。
>
> `No hits`（全期間でヒット0件）
> → 閾値が高すぎる。または `device_id.keyword` の値が間違っている（スペルミスなど）。
