## 全体方針

### なぜ「単一センサーから始めるか」

単一センサー（振動だけ）に絞ることで、ツールの複雑さに隠れていた本質が見えやすくなる。

```
❌ よくある失敗: 全部を同時に理解しようとして混乱する
✅ 正しいアプローチ: 1センサー × 1手法 を完全に理解 → 拡張する
```

「振動センサー1つで何ができるか」を完全に理解すれば、
温度・電流・圧力と増えても「同じことを繰り返すだけ」になる。

### アウトプット主体で進める

各フェーズに **必ず手を動かすアウトプット課題** を設定する。
「わかった気がする」と「自分でできる」は全く違う。
手を動かすことで本当に理解できているかがわかる。

### 「閾値をどう決めるか」を自分で考える力を育てる

このカリキュラムで特に重視するのは、**「閾値をどう設計するか」という思考力**。
ツールの使い方を覚えても、「どの値を使うか」「なぜその集計方法か」を
自分で判断できなければ、実務で応用できない。

```
❌ 与えられた閾値（例: vibration > 8.0）をそのまま使う
✅ 自分でデータを分析し、根拠をもって閾値を決める
   → 模範解答と比較し、なぜ違うのか・なぜ模範解答が優れているのかを理解する
```

Week 3 では必ず「自分の答えを先に決めてから」模範解答を見ること。
模範解答を先に見てしまうと、この学習の意味がなくなる。

## 1ヶ月の全体構成

```
Week 1（Day  1〜 7）: データの流れを「目で見て」理解し、正常値の特性を把握する
  → 単一センサー（振動）のデータを追い、正常・異常の統計を自分で調べる

Week 2（Day  8〜14）: 単一センサーで「異常を検知する」（ML アプローチ）
  → Elasticsearch ML で振動センサーの異常検知ジョブを自分で作る

Week 3（Day 15〜21）: 「閾値をどう決めるか」を考え、「異常をメールで通知する」
  ★ このカリキュラムの核心部分
  → 自分で閾値を設計し → 実装・テスト → 模範解答と比較・考察 → メール通知

Week 4（Day 22〜30）: 複数センサー・設備へ展開し、全体を統合する
  → センサー特性の違い・横展開の正しいやり方・全体ダッシュボードの統合
```

## 参考: 各ツール・概念の理解深度ガイド

### 1ヶ月後に「説明できるべきレベル」

| トピック | 理解の基準 |
|---|---|
| Logstash パイプライン | input/filter/output を自分で書き換えられる |
| Elasticsearch インデックス | keyword/float/date 型の使い分けを説明できる |
| Kibana Lens | 軸・集計方法・グラフ種類を自由に選べる |
| Elasticsearch ML | 教師なし異常検知の仕組みを図で説明できる |
| ElastAlert2 | ルールファイルを一から書けて、テストできる |
| アラート設計 | 閾値・ML・重複抑制の考え方を説明できる |
| センサー相関 | どの故障がどのセンサーで検知できるかを説明できる |
| 横展開 | ロジックを展開する方法と数値を使い回す危険性を説明できる |

### 「理解できた」の確認方法

> 「誰かに説明できるか？」が最も信頼できる確認方法。
> 技術的な背景を持たない人に説明できれば、本質的に理解している。
