### Elasticsearch ML のアルゴリズム

Elasticsearch ML の異常検知は、単一のアルゴリズムではなく **複数の技術を独自に融合したモデル** を使用している。

公式ドキュメントでは以下のように説明されている:

> "a bespoke amalgamation of different techniques"
> （複数の技術を独自に融合したもの）

具体的には、以下の4つの技術が組み合わされている:

```
┌────────────────────────────────────────────────┐
│          Elasticsearch ML のモデル構成          │
│                                                │
│  ① ベイジアン分布モデリング                     │
│     → データがどの分布に従うかを確率的に推定     │
│                                                │
│  ② 時系列分解（複数手法）                       │
│     → トレンド・季節性・残差に分解              │
│                                                │
│  ③ クラスタリング                               │
│     → 類似パターンをグループ化                  │
│                                                │
│  ④ 相関分析                                     │
│     → 複数フィールド間の関連性を捉える          │
│                                                │
│  これらを自動的に組み合わせて最適なモデルを構築  │
└────────────────────────────────────────────────┘
```

> ### コラム: なぜ具体的なアルゴリズム名が公開されていないのか
>
> Elasticsearch ML は商用製品（X-Pack）の一部であり、内部アルゴリズムの詳細は公開されていない。
> ただし重要なのは **「ユーザーがアルゴリズムを選ぶ必要がない」** という設計思想。
>
> 一般的な ML ツールでは「ランダムフォレストか SVM か LSTM か」をユーザーが選ぶ必要があるが、
> Elasticsearch ML では **分析関数（detector function）とデータ構造を指定するだけで、
> 最適なモデリング手法が自動選択** される。
>
> 公式ドキュメントの言葉を借りると:
> "You do not need to specify algorithms, models, or other data science-related configurations."
> （アルゴリズム、モデル、その他のデータサイエンス関連の設定を指定する必要はない）

### モデルが学習する3つの特徴

Elasticsearch ML のモデルは、時系列データから以下の3つの特徴を自動的に学習する。

#### 1. トレンド（傾向）

値が長期的に上昇・下降しているかを捉える。

```
例: センサー値が1ヶ月かけてゆっくり上昇している
    → モデルはこの上昇傾向を「正常な変化」として学習する

    ⚠️ Week 2 で学んだ「茹でガエル現象」の原因がこれ
    → 緩やかな劣化をトレンドとして取り込んでしまう
```

#### 2. 周期性（季節性）

時間帯や曜日による定期的な変動パターンを捉える。

```
例: 工場の稼働時間（8:00〜17:00）にセンサー値が高く、夜間は低い
    → モデルはこの日内パターンを学習する
    → 昼間に値が高いだけでは「異常」にならない（期待通りだから）
    → 夜間に値が高い場合は「異常」になる（期待と違うから）
```

#### 3. 値の分布

「どの値がどの程度の確率で出現するか」の分布を捉える。

```
例: 振動値が通常 2.8〜3.2 の範囲に 95% が収まっている
    → モデルはこの分布を学習する
    → 4.0 が出現すると「この値が出る確率は非常に低い」→ 異常スコアが高くなる
```

---

### 異常スコアの計算プロセス

Week 2 では「Actual と Typical の乖離」でスコアが決まると学んだ。ここではもう少し踏み込む。

```
Step 1: 各バケット（時間区間）ごとに「この値が出る確率」を計算
        → ベイジアン分布モデルが使われる
        → 確率が低いほど異常度が高い

Step 2: 生のスコアを集約（Aggregation）
        → 一時的なノイズを除去する

Step 3: スコアを正規化（Normalization）
        → 0〜100 のスケールに変換する
        → 異なるジョブ間でスコアを比較可能にする

Step 4: マルチバケット分析（Multi-bucket analysis）
        → 単一バケットでは軽微だが、連続して異常が出ている場合にスコアを引き上げる
        → 例: 1バケットだけ値が高い → スコア低め
             5バケット連続で値が高い → スコアを引き上げ
```

> ### コラム: マルチバケット分析とは
>
> 通常の異常検知は1バケット（1つの時間区間）単位で判定する。
> しかし現実には「1回だけ高い値が出た」よりも「何バケットも連続して高い値が出ている」方が
> 深刻な異常であることが多い。
>
> マルチバケット分析は **前後のバケットの異常度も考慮して、連続性のある異常のスコアを引き上げる** 機能。
>
> この機能はデフォルトで有効。ジョブ設定の `analysis_config.multivariate_by_fields` でも制御できるが、
> 通常はデフォルトのままで問題ない。

### オンライン学習の仕組み（復習 + 深掘り）

Week 2 で学んだオンライン学習をもう少し詳しく見る。

```
┌─────────────────────────────────────────────────┐
│              オンライン学習の動作                │
│                                                 │
│  初期学習（最初の数週間）                         │
│    └→ データの基本パターンを構築                  │
│                                                 │
│  継続学習（以降ずっと）                           │
│    └→ 新しいデータでモデルを常時更新              │
│    └→ 古いパターンは徐々に忘れていく              │
│    └→ 新しいパターンを取り込んでいく              │
│                                                 │
│  モデルスナップショット（3〜4時間ごとに自動保存）  │
│    └→ 特定時点のモデル状態に巻き戻し可能          │
│    └→ 「学習がおかしくなった」場合の保険           │
└─────────────────────────────────────────────────┘
```

**モデルスナップショットの活用**:
モデルが異常な期間のデータを「正常」として取り込んでしまった場合、
スナップショットを使って **異常期間の前の状態にモデルを巻き戻す** ことができる。

Kibana での操作:
1. **機械学習** → **異常検知** → **ジョブ管理**
2. 対象ジョブの **モデルスナップショット管理** を開く
3. 異常期間の前のスナップショットを選択して **「巻き戻す」**

## Day 34〜36: 分析関数を体系的に学ぶ

### 今日の目的

Elasticsearch ML の **分析関数（detector function）** を体系的に理解し、
「このデータにはどの関数を使うべきか」を判断できるようになる。

---

### 分析関数とは

分析関数は **「何を異常とみなすか」を決める設定**。
Week 2 では Mean（平均値）を使ったが、それ以外にも多くの選択肢がある。

```
ジョブ作成時に指定するもの:
  ① 分析関数（function）  → 「何を見るか」
  ② フィールド（field）   → 「どのデータを見るか」
  ③ split field          → 「どう分割するか」

例: high_mean(vibration) by device_id.keyword
     ①          ②            ③
  「vibration の平均が異常に高いケースを、設備ごとに検出する」
```

---

### 分析関数の分類

分析関数は大きく **7つのカテゴリ** に分類できる。

#### カテゴリ1: カウント系 — 「イベントの数」を見る

イベントの発生件数が普段と異なるかを検出する。**フィールド指定不要**（件数そのものを見るため）。

| 関数 | 検出方向 | 使いどころ |
|------|---------|-----------|
| `count` | 双方向（増加・減少の両方） | ログ量の監視全般 |
| `high_count` | 増加のみ | エラーログの急増、DDoS 攻撃 |
| `low_count` | 減少のみ | サービス停止（ログが出なくなった） |
| `non_zero_count` | 双方向（ゼロのバケットを無視） | まばらにしか発生しないイベント |
| `distinct_count` | 双方向 | ユニーク値の数が変わった（ポートスキャン等） |

**予知保全での活用例**:
```
使い方: アラートイベントの発生頻度を監視
  → count(alert_events) by device_id
  → 特定の設備からアラートが急増していないかを検出
```

---

#### カテゴリ2: メトリック系 — 「数値の大きさ」を見る ★ 予知保全で最も重要

数値フィールドの統計値が普段と異なるかを検出する。

| 関数 | 検出方向 | 使いどころ |
|------|---------|-----------|
| `mean` | 双方向 | センサー値の監視全般 |
| `high_mean` | 上昇のみ | 振動上昇、温度上昇の検出 |
| `low_mean` | 低下のみ | 圧力低下、流量低下の検出 |
| `median` | 双方向 | 外れ値の影響を受けにくい中心値の監視 |
| `min` | 双方向 | 最小値の異常（下限値の監視） |
| `max` | 双方向 | 最大値の異常（上限値の監視、瞬間的なスパイク検出） |
| `metric` | 双方向 | min, max, mean を一括で分析するショートカット |

**予知保全での使い分け**:
```
振動センサー:
  high_mean(vibration) → ベアリング劣化による振動上昇を検出
  max(vibration)       → 瞬間的な衝撃・異常振動のスパイクを検出

温度センサー:
  high_mean(temperature) → モーター過熱を検出
  low_mean(temperature)  → 冷却系統の停止を検出

電流センサー:
  mean(current) → 双方向の異常を検出（過負荷 or 空転）
```

> ### コラム: mean vs high_mean — 方向を指定する意味
>
> `mean` は値が上がっても下がっても異常として検出する。
> `high_mean` は値が上がったときだけ異常として検出する。
>
> **なぜ方向を指定するのか？**
>
> 例えば振動センサーの場合、設備を停止すると振動値は 0 に近くなる。
> `mean` を使うと、設備停止時に **「異常に低い」** という検出が発生してしまう。
> しかし設備停止は計画的なものであり、異常ではない。
>
> `high_mean` を使えば「上昇のみ」を検出するので、計画停止による誤検知を避けられる。
>
> **方向指定の判断基準**:
> - 上昇のみが問題 → `high_mean` / `high_count` / `high_sum`
> - 低下のみが問題 → `low_mean` / `low_count` / `low_sum`
> - どちらも問題 → `mean` / `count` / `sum`

---

#### カテゴリ3: 分散系 — 「ばらつき」を見る

値の安定性・変動性が変わったかを検出する。

| 関数 | 検出方向 | 使いどころ |
|------|---------|-----------|
| `varp` | 双方向 | データのばらつき変化全般 |
| `high_varp` | 増加のみ | 不安定化の検知（故障前兆で振動がばらつく） |
| `low_varp` | 減少のみ | 不自然な安定（センサー故障で値が固定） |

**予知保全での活用例**:
```
ベアリング劣化の初期段階:
  → 平均値はまだ正常範囲だが、ばらつきが増加する
  → high_varp(vibration) で平均値の変化より早く前兆を捉えられる可能性
```

> ### コラム: 「ばらつきの変化」は平均値の変化より早い？
>
> ベアリングが劣化し始めると、振動は「常に高い」のではなく
> 「時々大きく振れる」状態になることがある。
>
> ```
> 正常時:   2.9, 3.0, 3.1, 3.0, 2.9, 3.0  → 平均 3.0, ばらつき小
> 初期劣化: 2.8, 3.2, 2.7, 3.3, 2.9, 3.1  → 平均 3.0, ばらつき大 ← ここを捉える
> 進行劣化: 3.3, 3.5, 3.4, 3.6, 3.5, 3.7  → 平均 3.5, ばらつき大
> ```
>
> `varp` は平均値がまだ変わっていない段階でもばらつきの変化を検出できる。
> `mean` や `high_mean` と `high_varp` を **併用** することで、検知の網を広げられる。

---

#### カテゴリ4: 合計系 — 「合計値」を見る

フィールドの合計値が普段と異なるかを検出する。

| 関数 | 検出方向 | 使いどころ |
|------|---------|-----------|
| `sum` | 双方向 | 消費量・転送量の監視 |
| `high_sum` | 増加のみ | データ転送量の急増（情報漏洩検知） |
| `low_sum` | 減少のみ | 売上・生産量の急減 |
| `non_null_sum` | 双方向（null バケット無視） | まばらな金額データ |

予知保全での直接的な出番は少ないが、エネルギー消費量の監視などに使える。

---

#### カテゴリ5: 稀少値系 — 「めったに出ない値」を見る

通常は出現しない値が出現したことを検出する。

| 関数 | 特徴 | 使いどころ |
|------|------|-----------|
| `rare` | 時間的に稀な値を検出 | 新しいエラーコード出現、未知のステータスコード |
| `freq_rare` | 母集団内で稀な値へのアクセス頻度を検出 | 稀なファイルへの頻繁アクセス |

**制約**:
- `rare` は `by_field_name` の指定が必須
- `freq_rare` は `by_field_name` と `over_field_name` の両方が必須
- バケットスパンは **1時間未満** を推奨
- 最低 **20バケット分** の学習期間が必要

**予知保全での活用例**:
```
エラーコードの監視:
  rare(error_code) by device_id
  → 今まで出たことのないエラーコードが出現した → 新しい異常の可能性
```

---

#### カテゴリ6: 時間系 — 「いつもと違う時間」を見る

イベントが発生する時間帯のパターンの異常を検出する。

| 関数 | 特徴 | 使いどころ |
|------|------|-----------|
| `time_of_day` | 1日の中での異常な時間帯 | 深夜のログイン、営業時間外の操作 |
| `time_of_week` | 1週間の中での異常な曜日・時間帯 | 週末の業務アクセス |

バケットスパンは **10分程度** が推奨。予知保全よりもセキュリティ分析向け。

---

#### カテゴリ7: 特殊系

| 関数 | 特徴 | 使いどころ |
|------|------|-----------|
| `info_content` | 文字列の情報量（エントロピー）の異常 | DNS トンネリング、C2 通信の検出 |
| `lat_long` | 地理的位置パターンの異常 | クレジットカード不正利用（異常な場所での使用） |

これらはセキュリティ分析や不正検知向けのため、予知保全では通常使用しない。

### 予知保全で使う分析関数の選択ガイド

```
┌─────────────────────────────────────────────────────────────┐
│              予知保全での分析関数選択フロー                  │
│                                                             │
│  Q1: 何を監視したいか？                                      │
│                                                             │
│  ├─ センサー値の大きさ → Q2 へ                               │
│  ├─ センサー値のばらつき → varp / high_varp                  │
│  ├─ イベントの発生頻度 → count / high_count                  │
│  └─ 新しいエラーの出現 → rare                                │
│                                                             │
│  Q2: 値が上がるのが問題？下がるのが問題？両方？               │
│                                                             │
│  ├─ 上がるのが問題 → high_mean（振動↑、温度↑）              │
│  ├─ 下がるのが問題 → low_mean（圧力↓、流量↓）              │
│  └─ 両方問題 → mean（電流は上がっても下がっても問題）        │
│                                                             │
│  Q3: 瞬間的なスパイクも検出したい？                           │
│                                                             │
│  ├─ はい → max も追加                                        │
│  └─ いいえ → mean 系だけで十分                               │
│                                                             │
│  推奨: high_mean + high_varp の併用で検知範囲を最大化         │
└─────────────────────────────────────────────────────────────┘
```

### 実践: 分析関数を変えてジョブを作成する

Week 2 で作成した `sensor-anomaly-detection` ジョブは `Mean` を使っていた。
ここでは `high_mean` と `varp` を使ったジョブを作成し、結果を比較する。

#### ジョブ1: high_mean ジョブ

1. **機械学習** → **異常検知** → **ジョブを作成**
2. データビュー `sensor-data` を選択 → **マルチメトリック** を選択
3. **完全なデータを使用** → 次へ
4. メトリックに以下を追加:
   - `High Mean(vibration)`
   - `High Mean(temperature)`
   - `High Mean(current)`
5. 影響: `device_id.keyword`、バケットスパン: `15m`
6. ジョブID: `sensor-high-mean`
7. ジョブを作成

#### ジョブ2: varp ジョブ

このジョブはウィザードの **マルチメトリック** では作成できないため、**アドバンスド** ジョブで作成する。

1. **機械学習** → **異常検知** → **ジョブを作成**
2. データビュー `sensor-data` を選択 → **アドバンスド** を選択
3. detector を追加:
   - Function: `high_varp` / Field name: `vibration` / Partition field: `device_id.keyword`
   - Function: `high_varp` / Field name: `temperature` / Partition field: `device_id.keyword`
4. 影響: `device_id.keyword`
5. バケットスパン: `15m`
6. ジョブID: `sensor-varp`
7. ジョブを作成

## Day 37〜39: ジョブタイプと split field を使い分ける

### 今日の目的

4つのジョブタイプの違いと、split field（by / partition / over）の使い分けを理解する。

---

### 4つのジョブタイプ

Elasticsearch ML には 4 つのジョブタイプがある。

| ジョブタイプ | 比較対象 | 適した場面 | 作成方法 |
|---|---|---|---|
| **Single-metric** | 自分自身の過去データ | 1つのメトリックの監視 | ウィザード |
| **Multi-metric** | 自分自身の過去データ | 複数メトリックの同時監視 | ウィザード |
| **Population** | 母集団全体の平均的な挙動 | 多数の個体から異常な個体を発見 | ウィザード |
| **Advanced** | 設定による | 複雑な分析、複数 detector の組み合わせ | 手動設定 |

---

### Single-metric と Multi-metric（復習）

Week 2 で学んだ内容の整理。

```
Single-metric:
  - 1つのメトリック × 1つのジョブ
  - 例: vibration の平均値だけを監視
  - 最もシンプル。入門向け

Multi-metric:
  - 複数のメトリック × 1つのジョブ
  - 例: vibration, temperature, current を同時に監視
  - 現実の予知保全ではこちらが基本
```

---

### Population ジョブ — 「他と違う個体」を見つける

Population ジョブは、他のジョブタイプとは根本的に考え方が異なる。

```
通常のジョブ（Single / Multi / Advanced）:
  「この設備の過去のデータと比べて、今の値が異常か？」
  → 自分自身の履歴と比較

Population ジョブ:
  「他の設備と比べて、この設備の値が異常か？」
  → 母集団全体と比較
```

**図解**:

```
┌─ 通常のジョブ ──────────────────────┐
│                                      │
│  pump-01 の過去 ←比較→ pump-01 の今  │
│  pump-02 の過去 ←比較→ pump-02 の今  │
│  motor-01 の過去 ←比較→ motor-01 の今 │
│                                      │
│  各設備が独立して自分の過去と比較     │
└──────────────────────────────────────┘

┌─ Population ジョブ ────────────────────────┐
│                                             │
│  全設備の平均的な挙動 ←比較→ pump-01 の今   │
│  全設備の平均的な挙動 ←比較→ pump-02 の今   │
│  全設備の平均的な挙動 ←比較→ motor-01 の今  │
│                                             │
│  「他と比べておかしい個体」を見つける         │
└─────────────────────────────────────────────┘
```

**Population ジョブが向いている場面**:
- 同種の設備が **10台以上** ある（母集団のサンプル数が必要）
- 各設備の正常値が似ている
- 「他と比べておかしい設備」を一覧から見つけたい

**Population ジョブが向いていない場面**:
- 設備が **10台未満**（母集団が小さすぎる）
- 設備ごとに正常値が大きく異なる（pump と motor では振動の基準値が違う）

> ### コラム: Population ジョブの現実的な使い道
>
> 今回の PoC データは 3台しかないため、Population ジョブは本来不向き。
> しかし学習目的で作成し、通常ジョブとの結果の違いを体験する。
>
> 現実の工場で Population ジョブが威力を発揮するのは:
> - 同型のポンプが 50台並んでいるプラント
> - 同じ設計の風力発電タービンが 100基あるウィンドファーム
> - 同型の空調設備が 200台入ったデータセンター
>
> 「50台中、この1台だけ振動が高い」を一発で検出できる。

---

### 実践: Population ジョブを作成する

1. **機械学習** → **異常検知** → **ジョブを作成**
2. データビュー `sensor-data` を選択 → **母集団**（Population）を選択
3. **完全なデータを使用** → 次へ
4. Population field: `device_id.keyword`（これが母集団の分割基準）
5. メトリック: `Mean(vibration)`, `Mean(temperature)`, `Mean(current)` を追加
6. バケットスパン: `15m`
7. ジョブID: `sensor-population`
8. ジョブを作成

**結果の確認ポイント**:
```
□ 異常エクスプローラーで結果を確認する
□ 通常ジョブ（sensor-anomaly-detection）との結果の違いを観察する
□ 3台しかない環境で Population ジョブが有効に機能するか考察する
```

---

### split field の違い — by / partition / over

ジョブ設定で「設備ごとに分析する」方法は 3 つある。それぞれ意味が異なる。

| フィールド | 正式名 | 比較対象 | 典型的な使い方 |
|---|---|---|---|
| **by_field** | by_field_name | 各エンティティの **自分自身の過去** | 設備ごとの個別監視 |
| **partition_field** | partition_field_name | **グループ内** でモデルを分離 | 工場別・ライン別の分割 |
| **over_field** | over_field_name | **母集団全体** と比較 | Population ジョブ専用 |

**具体例で理解する**:

```
by_field_name: device_id.keyword
  → pump-01, pump-02, motor-01 それぞれに独立したモデルが作られる
  → pump-01 は pump-01 の過去データとだけ比較する
  → 結果: 「pump-01 は自分の過去と比べて振動が高い」

partition_field_name: device_id.keyword
  → by_field と似ているが、グループ間で情報を共有する
  → 新しい設備が追加されたとき、既存設備のモデルを参考にできる
  → 結果の見え方は by_field と似ている

over_field_name: device_id.keyword
  → 全設備の平均的な挙動と各設備を比較する
  → 結果: 「pump-01 は他の設備と比べて振動が高い」
```

> ### コラム: by_field と partition_field の違い（実践上の使い分け）
>
> 公式ドキュメントの説明だけでは違いがわかりにくいが、実践的には以下の使い分けが有効:
>
> - **by_field**: カーディナリティが低い場合（〜数十）に使用。
>   設備 ID、ホスト名など。
>
> - **partition_field**: カーディナリティが高い、またはグループ間で
>   情報を共有させたい場合に使用。
>   「pump-01 のモデルが、新しく追加された pump-03 の初期モデルに影響を与える」
>   というグループ間学習が行われる。
>
> 今回の PoC のように設備が 3台だけなら **by_field で十分**。
> 設備が 100台を超えるような環境では partition_field の方が
> メモリ効率・学習効率が良い場合がある。

## Day 40〜42: 精度チューニングと実践演習

### 今日の目的

「結果は出たが精度が悪い」ときに、何を調整すればいいかを学ぶ。
ML 異常検知の精度を上げるための 5 つのチューニング手法を体系的に習得する。

---

### 精度の問題は 2 種類ある

```
① 誤検知（False Positive）: 正常なのに異常と判定される
   → アラートが多すぎて信頼されなくなる（アラート疲れ）

② 検知漏れ（False Negative）: 異常なのに正常と判定される
   → 故障を見逃す

この 2 つはトレードオフの関係:
  精度を上げようとすると → 検知漏れが増える
  感度を上げようとすると → 誤検知が増える

チューニングの目的は「このバランスの最適点を見つけること」
```

---

### チューニング手法1: バケットスパンの調整

バケットスパンは **分析の時間粒度** を決める最も基本的な設定。

```
短いバケットスパン（5分）:
  ✅ 細かい変動を検出できる
  ✅ 異常検知の即時性が高い
  ❌ ノイズ（一時的な変動）も拾いやすい → 誤検知が増える
  ❌ リソース消費が大きい

長いバケットスパン（1時間）:
  ✅ ノイズが平滑化される → 誤検知が減る
  ✅ リソース消費が小さい
  ❌ 短時間の異常を見逃す
  ❌ 検知が遅れる
```

**選び方のガイドライン**:

| データの間隔 | 推奨バケットスパン | 理由 |
|---|---|---|
| 1秒〜1分 | 5分〜15分 | 高頻度データはノイズが多いので少し広く |
| 5分〜15分 | 15分〜30分 | データ間隔の 1〜3 倍程度 |
| 1時間 | 1時間〜2時間 | データ間隔と同じか 2 倍 |

**重要なルール**:
- 1日の周期で **割り切れる値** を選ぶ（15分, 30分, 1時間 など）
- 割り切れないと、日内の周期パターンの学習がうまくいかない

**今回の PoC データ**: 10分間隔 → `15m`（推奨）

---

### チューニング手法2: Influencer の最適化

Influencer は **「なぜ異常が起きたか」を特定しやすくする** 設定。
分析精度そのものには影響しないが、結果の解釈を大きく助ける。

```
Influencer に設定するフィールドの基準:
  ✅ 異常の原因になりうるフィールド
  ✅ カテゴリカル（離散的）なフィールド
  ❌ 数値フィールド（vibration, temperature 等）は不適

推奨: 3つ以下に絞る（多すぎるとリソース消費が増える）
```

**予知保全での推奨設定**:

```
Influencer 1: device_id.keyword   → 「どの設備で」異常が起きたか
Influencer 2: sensor_type.keyword → 「どのセンサーで」異常が起きたか（ある場合）
```

---

### チューニング手法3: カレンダーイベント（Scheduled Events）

**計画停止やメンテナンス期間** を事前に登録し、誤検知を防ぐ。

```
問題:
  設備を計画停止すると、センサー値が急に 0 になる
  → ML は「異常に低い」と検出してしまう（誤検知）

解決策:
  計画停止の期間を「カレンダーイベント」として登録
  → ML はその期間のデータを分析から除外する
```

**設定方法**（Kibana）:

1. **機械学習** → **設定**（Settings）
2. **カレンダー** → **新しいカレンダーを作成**
3. カレンダー名: `planned-maintenance`
4. ジョブID: 対象のジョブを選択
5. **新しいイベントを作成**:
   - 説明: `pump-01 定期メンテナンス`
   - 開始時刻: `2026-01-20 09:00`
   - 終了時刻: `2026-01-20 17:00`
6. 保存

> ### コラム: カレンダーイベントの注意点
>
> **重要**: カレンダーイベントは **データが分析される前に登録する必要がある**。
> 過去のデータに対して事後的にカレンダーイベントを登録しても効果がない。
>
> つまり:
> - リアルタイム分析の場合 → 計画停止の予定が決まった時点で登録
> - 過去データの再分析の場合 → ジョブ実行前に登録
>
> 今回のPoCでは過去データを扱っているため、ジョブを新規作成する前にカレンダーイベントを登録し、
> その後でジョブを作成・実行する必要がある。

---

### チューニング手法4: カスタムルール（Custom Rules）

特定の条件に合致するデータを **分析結果から除外** する。

```
例: device_id が "test-device" のデータは常に除外する

カスタムルールで設定:
  scope:
    device_id.keyword:
      filter_id: "test-devices"  # テスト設備のリスト
  actions: ["skip_result"]
```

**予知保全での活用例**:

```
① テスト運転中のデータを除外
② 特定のエラーコードを無視（既知の無害なエラー）
③ 閾値以下のスコアを無視（スコア 25 未満は興味なし）
```

**設定方法**（API）:

```bash
# フィルターの作成
curl.exe -u elastic:changeme -X PUT \
  "http://localhost:9200/_ml/filters/test-devices" \
  -H "Content-Type: application/json" \
  -d '{
    "description": "テスト設備リスト",
    "items": ["test-device-01", "test-device-02"]
  }'
```

ジョブの detector にカスタムルールを追加する場合は、ジョブ作成時に API で設定するか、
ジョブ作成後に API で更新する。

---

### チューニング手法5: 学習期間とモデルスナップショット

```
学習期間の考え方:
  ・最低 2 週間（日内パターン + 週内パターンを学習するため）
  ・推奨 4 週間以上（季節変動も捉えるなら 3 ヶ月以上）
  ・rare / freq_rare 関数は最低 20 バケット分が必要

モデルスナップショットの活用:
  ・3〜4 時間ごとに自動保存される
  ・「モデルが汚染された」場合に巻き戻しが可能
  ・例: メンテナンス期間のデータを学習してしまった
    → メンテナンス前のスナップショットに巻き戻す
    → ジョブを再実行する
```

---

### チューニングまとめ — 5 つの手法の使い分け

| 問題 | 対処法 | 手法 |
|---|---|---|
| 誤検知が多い（スコアが低い異常が大量に出る） | バケットスパンを長くする | 手法1 |
| 設備停止で誤検知が出る | カレンダーイベントで除外 | 手法3 |
| テスト設備のデータで誤検知が出る | カスタムルールで除外 | 手法4 |
| 検知漏れが多い（異常を見逃す） | バケットスパンを短くする、または varp を追加 | 手法1 |
| 異常の原因がわからない | Influencer を適切に設定 | 手法2 |
| モデルが異常パターンを学習してしまった | スナップショットで巻き戻し | 手法5 |

  □ 誤検知と検知漏れのバランスを調整する考え方を説明できる
```
