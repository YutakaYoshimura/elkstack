## 総合評価

**全体として非常に高品質なカリキュラムです。**

「先に考えてから模範解答を見る」という学習設計、コラムによる丁寧な概念説明、各 Day のアウトプット課題と理解確認チェックリスト、回答タイプ別フォローアップ（模範解答）の組み合わせは、初心者向けとして理想的な構成になっています。

ただし、学習者が実際に手を動かしたときに「詰まる原因」になる具体的な誤記・不整合がいくつかあります。以下に優先度別に列挙します。

---

## 優れている点（そのまま残すべき設計）

これらは修正不要。このカリキュラムの強みです。

1. **「先に考えてから模範解答を見る」設計**
   閾値設計の 3 問すべてに「Step 3 完了前に模範解答を開かない」という制約を設けている。強制的に思考を先行させる仕掛けが優れている。

2. **回答タイプ別フォローアップ（模範解答1〜3）**
   「8.0 にした場合」「平均値にした場合」「最大値にした場合」それぞれへの個別解説は、初心者教材として秀逸。読んだ人が「自分の考えのどこが問題だったか」を明確に理解できる。

3. **問題1→2→3 の設計（段階的な罠の仕掛け）**
   問題1（振動）で学んだ手法が問題2（温度）では通用しないケースを体験させ、問題3（横展開）で「数値コピーの危険性」を体験させる。この3問の構成は「自分で気づく」学習として非常に有効。

4. **各 Day のアウトプット課題とシートの具体性**
   「統計調査シート」「テスト結果シート」「比較・考察シート」など、記入すべき項目が具体的に定義されており、「わかった気がする」を防ぐ仕組みがある。

5. **コラムによる概念補足**
   `type: frequency`、`docker exec`、`extended_stats`、`avg+3σ` など、初心者が詰まりやすい概念に対してコラムが適切なタイミングで挿入されている。

6. **「横展開の正しい考え方」の表（Week 4）**
   「ロジックのコピー ✅ / 数値のコピー ❌」の対比表は明快で、実務的な視点として価値が高い。

### 指摘1: 設備名の誤記（Week 1）

**対象ファイル**: `02_Week1_データの流れを理解する.md`
**箇所**: Day 5〜7、パネル4 の説明

**現状**:
```
用途: pump-01、motor-01、compressor-01 の振動を重ねて比較する
```

**問題点**:
設備マスターに `compressor-01` は存在しません。正しくは **pump-01、pump-02、motor-01** の 3 台です。

学習者が Kibana で「全設備を重ねて表示」しようとしたとき、`compressor-01` のデータが存在しないため「自分の操作が間違っているのか」「データが足りないのか」と混乱します。

**修正案**:
```
用途: pump-01、pump-02、motor-01 の振動を重ねて比較する
```

### 指摘2: pump-01 の故障件数の誤記（00_提案.md）

**対象ファイル**: `00_提案.md`
**箇所**: Week 2 の Day 12〜14

**現状**:
```
pump-01 の3つの故障（1/20, 2/15）を全て Anomaly Explorer で確認する
```

**問題点**:
pump-01 の故障は **2 件**（1/20 ベアリング劣化 / 2/15 モーター過熱）です。3 件目（2/05 ベアリング劣化）は **motor-01** の故障です。「3つ」と「pump-01」の組み合わせが矛盾しており、Week 2 の Week 別ファイル（`03_Week2`）では「3 つの故障それぞれについて確認する」と正しく書かれているため、`00_提案.md` だけが誤っています。

**修正案**:
```
3 台・3 件の故障（pump-01:1/20・2/15、motor-01:2/5）を全て Anomaly Explorer で確認する
```

### 指摘4: `00_提案.md` と週別ファイルの役割が不明確

**対象ファイル**: `00_提案.md`、`02_Week1`〜`05_Week4`

**問題点**:
`00_提案.md` は Week 1〜4 の全詳細内容（Day ごとのやること・考えてほしい問い・アウトプット課題）を網羅しており、週別ファイルとほぼ同内容が重複しています。

学習者の視点では:
- 「`00_提案.md` を読んだ。週別ファイルは何のためにある？」
- 「`00_提案.md` と `02_Week1` で微妙に書き方が違うが、どちらに従えばよいか？」

という混乱が生じます。また `00_提案.md` には古い記述（誤ったファイルパス、旧ツール名 "Watcher" など）が残っており、週別ファイルとの齟齬の原因になっています。

**修正案（2択）**:
A. `00_提案.md` を「全体方針・概要・ロードマップのみ」に縮小し、Day 別の詳細は週別ファイルに一本化する（推奨）
B. 週別ファイルを廃止し、`00_提案.md` 1 ファイルに統合する（ファイル数を減らしたい場合）

### 指摘5: シェルコマンドのスタイルが混在

**対象ファイル**: 複数

|---|---|
| `00_提案.md` | bash スタイル（`\` で改行継続） |

### 指摘6: `anomaly-alert.yaml` テスト実行の前提条件が不明確

**対象ファイル**: `04_Week3_閾値設計とアラート通知.md`
**箇所**: Day 15〜16、テスト実行の手順

**問題点**:
`anomaly-alert.yaml` が監視する `prediction-results` インデックスは、Python ML の `batch_inference.py` を実行しないと存在しません。Step 1〜6 を一通り完了した状態でも、`batch_inference.py` を実行していなければこのインデックスは存在しません。

テスト実行結果が「0件（アラートなし）」になったとき、学習者は「操作が間違っているのか」「データがないのか」を判断できません。コラム中に「Python ML の batch_inference.py を実行した場合のみ prediction-results がある」と一文ありますが、気づかれにくい位置にあります。

**修正案**:
テスト実行の前に「前提確認」ステップを追加する:
```powershell
# prediction-results インデックスが存在するか確認
curl -u elastic:changeme "http://localhost:9200/prediction-results/_count?pretty"
# count が 0 または「index_not_found_exception」の場合は batch_inference.py を実行する
```
0件または存在しない場合の対処手順（`docker exec python-ml python batch_inference.py`）を明記する。

### 指摘7: `type: frequency` と `num_events` の説明に矛盾

**対象ファイル**: `模範解答/模範解答1_pump01振動_ベアリング劣化.md`
**箇所**: `num_events` と `timeframe` の解説

**問題点**:
本文では「10分間隔データで `timeframe: 30分, num_events: 3` にすると『30分継続』が条件になる」と推奨しているにもかかわらず、最終設計では `timeframe: 15分, num_events: 3` を採用しています。

10分間隔データで `timeframe: 15分` の場合、15分間に入るデータは最大 2 件（00分と10分）のみです。`num_events: 3` にすると「実質 25〜30 分継続」という条件になりますが、この計算と結論の関係が説明されていません。

「15分に3件 → 理論上は入らないが、実際にはバッファや処理タイミングにより入ることもある」という背景があるとすれば、それを補足する必要があります。

**修正案**:
最終設計の `timeframe: 15分, num_events: 3` について、「なぜ30分でなく15分を選んだか」の理由を一文追加する。または `timeframe: 30分, num_events: 3` に統一する。

### 指摘8: 「瞬間値 vs 直近平均」と実装（`type: frequency`）の概念乖離

**対象ファイル**: `問題/問題1_pump01振動_ベアリング劣化.md`、`01_目次.md`

**問題点**:
問題1の Step 2 では「瞬間値 vs 直近平均」という概念で閾値の監視方法を説明しています。「直近平均」は「直近 N 回の計測値の平均を計算して閾値と比較する」という意味に読めます。

しかし実際の ElastAlert2 での実装は `type: frequency`（指定時間内に閾値を超えた**件数**が N 以上）であり、「移動平均を計算して比較する」わけではありません。

この概念的な乖離により、学習者が「直近平均を計算して比較する実装 = `type: frequency`」と誤解する可能性があります。

また `01_目次.md` のチェックリストでは「「瞬間値」と「直近平均」どちらを使うべきか根拠を説明できる」と書かれていますが、`05_Week4` のチェックリストでは「「瞬間値」と「frequency（複数件）」のどちらを使うべきか根拠を説明できる」と正しく修正されています。`01_目次.md` の表現が古いままです。

**修正案**:
問題1の Step 2 の説明を以下のように修正する:

| 方法 | 内容 | 特徴 |
|---|---|---|
| 瞬間値（type: any） | 1 回でも閾値を超えたらアラート | ノイズに敏感。1件の突発値でも反応する |
| 複数件判定（type: frequency） | 指定時間内に N 件以上超えたらアラート | ノイズに強い。継続的な変化のみに反応する |

`01_目次.md` のチェックリストを `05_Week4` に合わせる。

### 指摘9: 正規分布の前提確認ステップがない

**対象ファイル**: `問題/問題2_pump01温度_モーター過熱.md`、`模範解答/模範解答2_pump01温度_モーター過熱.md`

**問題点**:
`avg + 3σ` の閾値設計は「データが正規分布（または近似）に従う」ことを前提にしています。問題1の振動センサーでは問題なく機能しますが、問題2の温度センサーは電流や時間帯の影響を受けるため、データが二峰性分布（昼高・夜低）を示す可能性があります。

問題2の Step 0 で「昼間と夜間で温度の水準が違うか確認する」手順はあります。しかし「なぜそれが問題になるか」の論理的な説明が弱く、「時間帯変動があると `avg + 3σ` の精度が下がる理由」が明示されていません。

模範解答2の冒頭では「avg+3σ の閾値（約 55.9）では正常な高温にも反応してしまう」と書かれているだけで、「なぜ avg+3σ が低めに出るか」の理由（夜間の低温データが平均を引き下げるため）が説明されていません。

**修正案**:
模範解答2に以下の説明を追加する:
「温度センサーのデータに昼夜差がある場合、夜間の低温データが avg を引き下げる。結果として算出される avg+3σ は『夜間基準の閾値』になり、昼間の正常な高温（稼働中）でも閾値を超えてしまう。これが誤検知の原因。」
